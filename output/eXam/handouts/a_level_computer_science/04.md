## Von Neumann architecture

The **Von Neumann model** is the design pattern underlying almost every general-purpose computer. Its key features:

- A **single memory** holds both **program instructions** and **data** (the "stored program" concept).
- A **processor (CPU)** that fetches instructions from memory and executes them one at a time.
- **Sequential execution** — instructions are executed in order unless a branch instruction changes the flow.

The stored-program concept is what makes a computer flexible: changing the program changes what the computer does, with no need to rewire the hardware.

## The CPU's main components

### Arithmetic and Logic Unit (ALU)

Performs the **arithmetic operations** (add, subtract, multiply, divide) and **logical operations** (AND, OR, NOT, comparisons) on data. Receives operands from registers, produces results that are usually placed back in a register.

### Control Unit (CU)

**Decodes** each fetched instruction and **issues the control signals** needed to carry it out — opening and closing data paths between registers, telling the ALU which operation to perform, controlling memory reads and writes.

### System clock

Generates a **steady stream of pulses** that synchronise the operations of the CPU. Each instruction takes a fixed number of clock cycles. **Clock speed** (e.g. 3.8 GHz) is one factor in performance.

### Immediate Access Store (IAS)

The CPU's directly accessible memory — typically the **main memory** (RAM). Holds the program currently running and its data. Access time is short compared with secondary storage but slow compared with registers.

### Registers

**Registers** are very small, very fast storage locations inside the CPU. Two categories:

**Special-purpose registers** — each has a defined role in the F-E cycle:

- **Program Counter (PC)** — holds the address of the **next** instruction to be fetched.
- **Memory Address Register (MAR)** — holds the address currently being read or written.
- **Memory Data Register (MDR)** — holds the data being transferred to or from memory.
- **Current Instruction Register (CIR)** — holds the instruction currently being decoded and executed.
- **Accumulator (ACC)** — holds the value the ALU is currently working on; arithmetic and logic operations typically read one operand from and write the result to the ACC.
- **Status Register** — holds **flags** that record the state of the last operation (carry, zero, negative, overflow), used by branch instructions.
- **Index Register** — holds an offset used in addressing modes (see "Indexed addressing").

**General-purpose registers** — used by the programmer for temporary storage of operands and intermediate results during a calculation. A modern CPU has many of these (e.g. 16 in x86-64).

## Buses

Components of the computer system are connected by three internal buses, each a set of parallel wires:

- **Address bus** — carries the memory **address** of the location being read or written. **One-way**: from the CPU to memory and I/O.
- **Data bus** — carries the **data** being read from or written to memory or I/O. **Two-way**: data flows both directions.
- **Control bus** — carries **control signals** (read, write, interrupt, clock pulses) between components. **Two-way** (different signals go each way).

The width of the address bus determines how much memory can be addressed: an $n$-bit address bus can address $2^{n}$ memory locations. The width of the data bus determines how many bits move per memory access (often equal to the word size of the architecture, e.g. 32 or 64 bits).

## What affects performance

Factors that change how fast the computer system runs:

- **Clock speed** — higher clock means more cycles per second; each instruction completes faster.
- **Number of cores** — a multi-core CPU can run several threads in parallel. A **quad-core** does up to 4× the work of a single core for parallelisable workloads.
- **Word size / architecture width** — a **32-bit** CPU processes 32-bit chunks per cycle; a **64-bit** CPU processes 64-bit chunks. Wider word = more work per cycle (especially for large integers, addresses, and certain media tasks). 64-bit also lets you address far more memory ($2^{64}$ bytes vs $2^{32}$ bytes ≈ 4 GiB).
- **Amount of RAM** — more RAM means more of the working program and data can be held in fast main memory. Too little RAM forces the OS to **page** to disk, slowing everything down.
- **Cache size / levels** — larger and more levels of cache reduce average memory access time.
- **Secondary-storage type** — SSDs are far faster than magnetic HDDs at random reads/writes, so the system starts up faster, programs load faster.
- **Bus width and speed** — wider/faster buses move data between components more quickly.

When comparing two computers, look at how their specs match the intended workload. A faster clock at narrower architecture may lose to a slower clock at wider architecture for memory-bound work; a quad-core may beat a dual-core for parallel work, but if the workload is single-threaded the higher per-core speed wins.

## Ports

A **port** is a physical interface on a computer that lets it connect to peripheral devices:

- **USB** — Universal Serial Bus. Standard general-purpose port for keyboards, mice, external drives, printers, phones. Has multiple generations and connector shapes (USB-A, USB-C).
- **HDMI** — High-Definition Multimedia Interface. Carries digital video and audio to monitors and TVs.
- **VGA** / **DVI** / **DisplayPort** — other display interfaces.
- **Ethernet (RJ-45)** — wired LAN.
- **Audio jacks** — 3.5 mm for headphones / microphones; some use S/PDIF for digital audio.
- **Power input** — for the device's own power supply.

Different ports use different signalling and protocols; you cannot plug an HDMI cable into a USB socket. Modern USB-C is unusual in that it can carry video, data, and power.

## Fetch-Execute (F-E) cycle

The CPU repeats the following cycle continuously, one execution per machine instruction.

### Fetch

1. The address in the **PC** is copied to the **MAR**.
2. The PC is **incremented** by 1 (or by the instruction length) to point to the next instruction.
3. A **read** signal is sent over the control bus to memory.
4. Memory places the instruction at that address on the **data bus**.
5. The instruction is copied from the data bus into the **MDR**.
6. The MDR's contents are then copied into the **CIR**.

### Decode

The **CU** decodes the instruction in the CIR — works out what operation is to be performed and what operands or addresses it needs.

### Execute

The CU sends control signals to carry out the operation:

- arithmetic / logic operations are handed to the **ALU**, with operands from registers or memory; the result goes to the ACC or another register.
- a load or store fetches data from memory into a register or vice versa.
- a branch instruction modifies the PC, jumping to a different part of the program.

Then the cycle repeats from "fetch".

## Interrupts

An **interrupt** is a signal that **temporarily suspends** the normal F-E cycle so the CPU can handle an event that needs attention.

Sources of interrupts:

- **I/O devices** — a key has been pressed, a packet has arrived, a disk read has completed.
- **Hardware faults** — power failing, hardware error.
- **Software** — division by zero, illegal instruction.
- **Timer** — used by the OS to schedule processes.

### Handling an interrupt

1. The CPU finishes the current instruction.
2. It **saves its state** (PC and registers) so it can resume later.
3. It loads the address of the appropriate **interrupt service routine (ISR)** into the PC and starts executing the ISR.
4. The ISR handles the event (e.g. reads the keystroke into a buffer).
5. The CPU **restores** the saved state and resumes the previous program from where it left off.

Interrupts are essential — they let the system **respond promptly** to I/O without the CPU having to poll devices constantly. They are how the OS multitasks and how peripherals "wake up" the processor.

## Assembly language and machine code

### Machine code

The instructions a CPU actually executes are **machine code** — strings of bits where each instruction encodes an operation and its operands. Specific to one CPU architecture; not portable.

### Assembly language

**Assembly language** is a human-readable form of machine code. Each assembly instruction normally maps to **one machine instruction**. Uses **mnemonics** like `LDD`, `ADD`, `STO`, `JMP`. Easier to read and write than raw binary, but still architecture-specific.

An **assembler** translates assembly into machine code. Each line of assembly becomes one machine instruction (with the same operands).

### Two-pass assembler

A common assembler design makes **two passes** over the source code:

**Pass 1** — build the symbol table:

- Go through the source line by line.
- Each time a **label** is encountered (e.g. `LOOP:`), record its address (its position in the output) in a symbol table.
- Don't generate code yet.

**Pass 2** — generate code:

- Go through the source line by line again.
- Translate each instruction to its machine-code form.
- When an instruction references a label (e.g. `JMP LOOP`), look up the label's address in the symbol table and substitute it.

The two-pass approach handles **forward references** (a jump to a label defined later in the code).

## Instruction groupings and an example instruction set

Cambridge assembly questions typically use a small, generic instruction set, organised into groups:

- **Data movement**: `LDD` (load direct), `LDM` (load immediate), `LDI` (load indirect), `LDX` (load indexed), `STO` (store), `MOV` (register to register).
- **Arithmetic**: `ADD`, `SUB`, `INC` (increment), `DEC` (decrement).
- **Logic / bit manipulation**: `AND`, `OR`, `XOR`, `LSL` (logical shift left), `LSR` (logical shift right).
- **Compare and branch**: `CMP`, `JMP` (unconditional jump), `JPE` (jump if equal), `JPN` (jump if not equal).
- **Input / output**: `IN` (read from input), `OUT` (write to output).
- **End**: `END`.

The exact mnemonics in an exam paper are usually given in a reference table — read it carefully.

## Addressing modes

The **addressing mode** of an instruction specifies how the CPU finds the operand. Common modes:

- **Immediate** — the operand is the value **in the instruction itself**. `LDM #10` loads the literal number 10 into the ACC.
- **Direct** — the instruction contains an **address**; the operand is the value **at that address**. `LDD 200` loads the value stored at memory location 200.
- **Indirect** — the instruction contains an address that holds **another address**, which is the actual data location. `LDI 200` loads the value at the address that is itself stored at location 200.
- **Indexed** — the instruction contains an address plus the contents of an **index register**. The effective address is `(address + IR)`. Used to step through arrays. `LDX 100` with IR = 5 loads the value at address 105.
- **Relative** — the address is **relative to the current PC**. Useful for position-independent code (not always tested at A-Level).

## Tracing an assembly program

To **trace** an assembly program:

1. Set up a table with columns for the PC, ACC, IR (if used), each variable, and any other registers/flags used by the program.
2. Walk through the instructions in order. After each instruction, update the values in the table.
3. Branches change the PC — follow them.
4. End when an `END` instruction is reached (or when the trace question asks you to stop).

A common pattern: an assembly loop that processes an array using indexed addressing — initialise the index, do the work, increment the index, compare to the upper bound, jump back if not done.

## Bit manipulation: binary shifts

### Logical shifts

A **logical shift** moves all the bits of a binary number left or right by a given number of places, **filling new positions with 0**.

- **Left shift by 1** (`LSL #1`) — every bit moves one place left; a 0 enters the rightmost position; the leftmost bit falls off. For an unsigned number this is **multiplication by 2** (provided no overflow).
- **Right shift by 1** (`LSR #1`) — every bit moves one place right; a 0 enters the leftmost position; the rightmost bit falls off. For an unsigned number this is **integer division by 2** (the discarded bit is the remainder).

Shifting by $n$ places multiplies (or divides) by $2^{n}$ for unsigned numbers.

Examples (8-bit unsigned):

- `00001011` (= 11) `LSL #1` → `00010110` (= 22).
- `01000000` (= 64) `LSR #2` → `00010000` (= 16).

### Arithmetic shifts (less common in this syllabus)

An **arithmetic right shift** preserves the sign bit (the leftmost bit) so that a signed negative number stays negative after the shift. Not usually required at this level beyond mention.

### Bit manipulation to monitor/control a device

Embedded devices commonly use individual bits in a register to represent independent signals. Each bit corresponds to one sensor or actuator — for example, in an 8-bit register controlling a row of LEDs:

- bit 0 (rightmost) = LED 0 on/off,
- bit 1 = LED 1 on/off,
- ...
- bit 7 = LED 7 on/off.

To **set** bit $n$ (turn it on): OR the register with a mask that has bit $n$ set and all others clear (`R = R OR 0001 0000` to set bit 4).

To **clear** bit $n$ (turn it off): AND the register with a mask that has bit $n$ clear and all others set (`R = R AND 1110 1111` to clear bit 4).

To **toggle** bit $n$: XOR the register with a mask that has bit $n$ set (`R = R XOR 0001 0000`).

To **test** whether bit $n$ is set: AND the register with the mask and check whether the result is non-zero (for that one bit).

Bit manipulation is fast and uses little memory — important on small embedded systems. It also lets a single byte represent a vector of up to 8 binary states (sensor readings, status flags, control lines) compactly.
