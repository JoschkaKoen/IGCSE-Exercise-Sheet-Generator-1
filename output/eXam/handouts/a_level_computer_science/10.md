## Choosing data types

Every variable needs a **data type** — the kind of value it holds and the operations allowed on it. The standard pseudocode data types are:

- `INTEGER` — whole number (positive, negative, or zero). E.g. `42`, `-7`. Use for counts, indexes, IDs.
- `REAL` — a number with a fractional part. E.g. `3.14`, `-0.5`, `12.67`. Use for money, measurements, averages.
- `STRING` — a sequence of characters in quotes. E.g. `"Hello"`, `"Fruit"`. Use for text.
- `CHAR` — a single character in quotes. E.g. `'A'`, `'?'`. Use when only one character matters.
- `BOOLEAN` — `TRUE` or `FALSE`. Use for flags, yes/no answers, conditions.
- `DATE` — a calendar date. E.g. `20/02/2025`.

Pick the smallest, most precise type that fits. Use `INTEGER` for whole counts (not `REAL`); use `BOOLEAN` for flags (not the strings `"yes"`/`"no"`).

## Records

A **record** is a single composite structure that holds **several fields of different data types** under one identifier. Records are useful when several pieces of data describe one logical thing.

### Defining a record in pseudocode

```
TYPE TStockItem
    DECLARE ItemID : INTEGER
    DECLARE Category : STRING
    DECLARE ItemCost : REAL
    DECLARE InStock : BOOLEAN
ENDTYPE
```

This defines the **type** `TStockItem`. To use it, declare variables of that type:

```
DECLARE Item1 : TStockItem
DECLARE Items : ARRAY[1:100] OF TStockItem
```

### Reading and writing record fields

Use dot notation to access each field:

```
Item1.ItemID ← 42
Item1.Category ← "Fruit"
Item1.ItemCost ← 12.67
Item1.InStock ← TRUE

OUTPUT Item1.Category, " costs ", Item1.ItemCost
```

A record can be read from input or written to output by handling each field individually, or — for many languages and pseudocode — by passing the whole record to a function.

### When to use a record

Use a record when a set of values **always belong together** as one entity (a customer, a stock item, a transaction). Use **separate variables** for unrelated values.

## Arrays

An **array** is an ordered collection of items of the **same data type**, stored under one identifier, accessed by an **index**.

### Terminology

- **Element** — one item stored in the array.
- **Index** (or **subscript**) — the position of an element. Cambridge pseudocode uses index ranges you specify explicitly, e.g. `1:n`.
- **Bounds** — the lowest and highest valid indices.
- **Length / size** — the number of elements (upper bound − lower bound + 1).
- **Dimension** — 1-D (a list), 2-D (a table of rows and columns), 3-D, etc.

### 1-D arrays

```
DECLARE Names : ARRAY[1:5] OF STRING

Names[1] ← "Ada"
Names[2] ← "Bob"
Names[3] ← "Cara"
Names[4] ← "Dan"
Names[5] ← "Eve"

OUTPUT Names[3]   // outputs "Cara"
```

Process every element with a `FOR` loop:

```
FOR i ← 1 TO 5
    OUTPUT Names[i]
NEXT i
```

### 2-D arrays

```
DECLARE Grid : ARRAY[1:3, 1:4] OF INTEGER

Grid[2, 3] ← 99
```

The first index is the row, the second the column. Process every cell with nested loops:

```
FOR row ← 1 TO 3
    FOR col ← 1 TO 4
        OUTPUT Grid[row, col]
    NEXT col
NEXT row
```

### Choosing 1-D vs 2-D

- **1-D** when the data is a single sequence (a list of names, a queue, a single row of measurements).
- **2-D** when the data has two natural dimensions (a grid, a matrix, rows × columns, days × hours).

### Common array operations

**Search**: linear search through every element until found (or end reached):

```
FOR i ← 1 TO n
    IF A[i] = Target THEN
        OUTPUT "Found at ", i
        // and probably break out of the loop
    ENDIF
NEXT i
```

**Sum / count / maximum / minimum** — initialise a running variable then sweep through:

```
Max ← A[1]
FOR i ← 2 TO n
    IF A[i] > Max THEN
        Max ← A[i]
    ENDIF
NEXT i
```

**Filter / copy** — copy elements meeting a condition into a new array, keeping a separate destination index.

**Sort** — bubble sort, insertion sort, quicksort — covered in the algorithms topic.

## Files

A **file** is a stored collection of data on **secondary storage**, persisted between program runs.

### Why files are needed

Variables in RAM disappear when the program ends. To save data permanently (high-score tables, customer records, log files, settings), the program must write to a file. Files also let:

- different programs share data,
- the program restart from a previous state,
- the user back up, copy or move the data.

### Text files

A **text file** consists of lines of human-readable characters, separated by line-ending characters. Pseudocode operations on text files:

**Open a file** for reading or writing — must be done before any read/write, and **closed** afterwards.

```
OPENFILE "data.txt" FOR READ      // read existing
OPENFILE "data.txt" FOR WRITE     // create / overwrite
OPENFILE "data.txt" FOR APPEND    // add to end
```

**Read** the next line into a string variable:

```
READFILE "data.txt", LineString
```

**Test end of file** before reading:

```
WHILE NOT EOF("data.txt") DO
    READFILE "data.txt", LineString
    OUTPUT LineString
ENDWHILE
```

**Write** one line:

```
WRITEFILE "data.txt", "Hello world"
```

**Close** the file:

```
CLOSEFILE "data.txt"
```

### Reading a file as a whole

A common pattern: open, loop until EOF, do something with each line, then close.

```
OPENFILE "stock.txt" FOR READ
WHILE NOT EOF("stock.txt") DO
    READFILE "stock.txt", line
    OUTPUT line
ENDWHILE
CLOSEFILE "stock.txt"
```

### Writing a file

```
OPENFILE "log.txt" FOR WRITE
FOR i ← 1 TO 100
    WRITEFILE "log.txt", "Event " & i
NEXT i
CLOSEFILE "log.txt"
```

Always **close** every file you open. If you don't, data may be lost (writes are sometimes buffered) and other programs may not be able to use the file.

## Abstract Data Types (ADTs)

An **Abstract Data Type** is a **collection of data and a set of operations on that data**, defined by **what** it does — not how it's stored internally.

The user of an ADT works only through the defined operations; the implementation details are hidden. This separation lets the implementation be changed without affecting code that uses the ADT.

Three ADTs you should know: **stack**, **queue**, **linked list**.

### Stack

A **stack** stores data in **LIFO** order — **Last In, First Out**. The most recently added item is the first to be removed.

Operations:

- `Push(x)` — add `x` to the **top** of the stack.
- `Pop()` — remove and return the **top** item.
- `Peek()` (or `Top()`) — return the top item without removing it.
- `IsEmpty()` — true if the stack has no items.
- `IsFull()` — true if it cannot accept another item (for fixed-size implementations).

Uses: undo history, function-call return addresses, parsing expressions, reversing a sequence, backtracking algorithms.

### Queue

A **queue** stores data in **FIFO** order — **First In, First Out**. The longest-waiting item is the first to be removed.

Operations:

- `Enqueue(x)` (or `Add`) — add `x` to the **rear**.
- `Dequeue()` (or `Remove`) — remove and return the item at the **front**.
- `IsEmpty()`, `IsFull()`.

Uses: print spooling (jobs are served in the order they arrive), waiting customers, breadth-first search, scheduling, buffering between producer and consumer.

### Linked list

A **linked list** stores data as a sequence of **nodes**. Each node holds a **value** and a **pointer** (or "next" reference) to the next node. A head pointer marks the start; the last node's "next" pointer is a sentinel value (e.g. `-1`, `NULL`) marking the end.

Operations:

- Insert at front / at end / before-or-after a given node.
- Delete a node.
- Search for a value.
- Traverse the list to visit each node in order.

Uses: dynamic lists that grow and shrink, ordered collections where insertion order matters, implementing stacks/queues of unbounded size.

The advantage over an array is that **insertion and deletion** are cheap (just adjust pointers — no need to shift elements). The disadvantage is that **random access** is slow (you have to follow pointers from the head).

## Implementing ADTs using arrays

ADTs can be implemented in many ways. For each, you should be able to describe an **array-based implementation**.

### Stack using an array

Hold the items in an array `Stack[1:MaxSize]`, with an integer `Top` indicating the index of the current top item (or 0 when empty).

- `Push(x)`:
  - if `Top = MaxSize`, the stack is **full** — overflow.
  - else `Top ← Top + 1`; `Stack[Top] ← x`.
- `Pop()`:
  - if `Top = 0`, the stack is **empty** — underflow.
  - else `value ← Stack[Top]`; `Top ← Top - 1`; return `value`.

### Queue using an array (circular buffer)

A naïve implementation: hold items in an array `Queue[1:MaxSize]` with two pointers `Front` and `Rear`. Enqueue adds at `Rear` and increments `Rear`; dequeue takes from `Front` and increments `Front`.

Problem: the pointers march off the end of the array, leaving empty space at the start unused. Solution: a **circular array** — when `Front` or `Rear` reaches `MaxSize`, it wraps back to 1.

- `Enqueue(x)`:
  - check `IsFull` (the queue is full when `Rear+1 mod MaxSize = Front`).
  - else `Rear ← (Rear MOD MaxSize) + 1`; `Queue[Rear] ← x`.
- `Dequeue()`:
  - check `IsEmpty`.
  - else `value ← Queue[Front]`; `Front ← (Front MOD MaxSize) + 1`; return `value`.

You also need to track whether the queue is empty or full, e.g. with a separate item count.

### Linked list using an array

Use two parallel arrays (or one array of records):

```
TYPE TNode
    DECLARE Value : <whatever the list stores>
    DECLARE Next : INTEGER     // index of the next node, or -1 for end
ENDTYPE

DECLARE Nodes : ARRAY[1:MaxSize] OF TNode
DECLARE Head : INTEGER         // index of first node, -1 if list is empty
DECLARE FreeListHead : INTEGER // index of the first available free node
```

`FreeListHead` chains together the unused slots in the same way the data list chains its used ones. To insert a new node:

1. Take a free slot index from `FreeListHead`; advance `FreeListHead` to `Nodes[FreeListHead].Next`.
2. Set the new node's `Value` and `Next` (pointing to where it should sit in the data list).
3. Update the previous node's `Next` (or `Head`) to point to the new node's index.

To delete a node, do the reverse: unlink it from the data list, and put its index at the front of the free list.

The array-based linked list combines the dynamism of a linked structure with the static allocation of an array — useful in environments without dynamic memory.
