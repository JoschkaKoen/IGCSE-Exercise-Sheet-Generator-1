## Why protocols are needed

A **protocol** is a **set of rules** governing how two or more computing devices **communicate**. Both ends must follow the same rules; otherwise the signals one side sends are unintelligible to the other.

Protocols define:

- **the format** of the data being sent (where the addresses sit, where the payload starts, how errors are reported),
- **the order** of messages (who speaks first, when to acknowledge, when to retransmit),
- **the meaning** of each kind of message,
- **the timing** of messages (timeouts, retransmits, keepalives),
- **what to do when things go wrong** (lost packets, invalid data).

Without an agreed protocol, communication would not work at all — like two people speaking different languages with no translator.

## Layered protocols (the stack model)

Networking is **complex**. Rather than designing one giant protocol that handles every detail, networking is divided into **layers**. Each layer has a focused responsibility and talks **only to the layer above and the layer below**.

Benefits:

- **Modularity** — a layer can be modified or replaced without touching the others (e.g. swap Ethernet for Wi-Fi at the bottom; everything above is unchanged).
- **Standardisation** — each layer has well-defined interfaces; different vendors' implementations interoperate.
- **Abstraction** — programmers writing at one level don't worry about details handled by other layers.

The two most-used layered models in this syllabus are the OSI model (7 layers, not required in detail) and the **TCP/IP suite** (4 layers).

## TCP/IP protocol suite

The internet runs on the **TCP/IP** suite, with four layers (top to bottom):

| Layer | Purpose | Examples |
|---|---|---|
| **Application** | end-user applications — what the program does | HTTP, FTP, SMTP, POP3, IMAP, BitTorrent |
| **Transport** | end-to-end delivery of data between processes | TCP, UDP |
| **Internet** (Network) | routing of packets between networks | IP, ICMP |
| **Link** (Network access) | sending bits over a physical medium | Ethernet, Wi-Fi |

### Application layer

Provides services directly to user-facing programs and defines the protocols those programs speak (HTTP for web, SMTP for sending email, etc.). The layer at which a programmer most commonly works — the underlying details (sockets, packets, frames) are handled by the lower layers.

### Transport layer

Provides **end-to-end** delivery of data between application-layer processes — typically identified by **port numbers**.

Two main protocols:

- **TCP (Transmission Control Protocol)** — **connection-oriented**. Establishes a connection, ensures **all data arrives in order**, retransmits lost packets, controls flow. Reliable but adds overhead. Used by HTTP, HTTPS, SMTP, FTP, IMAP, POP3.
- **UDP (User Datagram Protocol)** — **connectionless**. Sends data and forgets — no acknowledgements, no retransmission, no ordering. Lower overhead, but no reliability guarantees. Used for streaming media, DNS, online gaming where speed beats reliability.

The transport layer also segments large messages into smaller pieces that fit into the maximum packet size, and reassembles them at the destination.

### Internet layer

Carries packets between hosts across one or more networks. The main protocol is **IP** (Internet Protocol). Each packet has a source IP address and a destination IP address. Routers in the path forward each packet towards the destination.

The Internet layer does not guarantee delivery — that's the job of TCP at the transport layer.

### Link layer

Concerned with sending bits over a single physical link — twisted-pair Ethernet, Wi-Fi, fibre, cellular. Adds and removes a frame header carrying MAC addresses and a frame check sequence. Handles physical-medium specifics like access control (CSMA/CD on Ethernet) and modulation.

## Common application-layer protocols

You should know what each does and on which type of connection it's used.

### HTTP (HyperText Transfer Protocol)

Used by **web browsers** to fetch web pages and other resources from web servers. Stateless request-response over TCP, usually on port 80. The client sends a request (`GET /page.html HTTP/1.1`); the server responds with the resource. **HTTPS** is HTTP over TLS/SSL — encrypted, on port 443.

### FTP (File Transfer Protocol)

Used to **transfer files** between a client and a server over a network. Supports browsing directories, uploading, downloading. Uses TCP. Has a control connection and a data connection. Less common today (often replaced by HTTPS, SFTP, or cloud storage), but still in use.

### SMTP (Simple Mail Transfer Protocol)

Used to **send email** — from a mail client to a mail server, and between mail servers. Over TCP, port 25 (or 465/587 for secure variants). SMTP only **sends**; receiving requires POP3 or IMAP.

### POP3 (Post Office Protocol v3)

Used by a mail client to **retrieve email** from a mail server. Traditionally **downloads** messages and then **deletes** them from the server, so they exist only on the client. Simple, lightweight; not well-suited if you read mail on multiple devices.

### IMAP (Internet Message Access Protocol)

Also used to retrieve email — but **leaves the messages on the server** and synchronises folder/state across multiple clients. Better than POP3 when you want to read your mail on a phone, a laptop and a desktop and see the same inbox everywhere.

### BitTorrent

A **peer-to-peer** file-sharing protocol. A file is split into many small pieces. Each piece is downloaded **from many peers in parallel** — no single central server bears all the bandwidth load. The more peers that have the file, the faster the download. Used (legitimately) to distribute large open-source software releases, video, scientific data; also (illegitimately) for piracy.

## Circuit switching vs packet switching

When two devices want to exchange data over a network, two general approaches exist for routing the data: **circuit switching** and **packet switching**.

### Circuit switching

A **dedicated path** (a "circuit") is set up between the two endpoints before any data is sent. The circuit is reserved for the duration of the conversation. Data flows along it in order, at a constant rate.

- **Setup phase**: a path is established, reserving capacity at each switch along the way.
- **Conversation phase**: data flows along the fixed path.
- **Teardown phase**: when the conversation ends, the path is released.

Characteristics:

- **Reserved bandwidth** — the connection has a fixed quality and predictable performance.
- **In-order delivery** — the path is fixed, so packets arrive in the order they left.
- **Inefficient** during silent periods — bandwidth is reserved even when no data is being sent.
- **Slow to set up** — establishing the circuit takes time.

Classic example: the **traditional public telephone network**. When you dialled a call, the telephone exchanges set up a circuit from your phone to the other phone, reserved that path for the call, and tore it down when you hung up.

### Packet switching

The data is broken into small **packets**. Each packet is sent **independently** across the network. Each packet carries the **destination address** in its header; **routers** make per-packet routing decisions, forwarding each packet along the current best path. Packets may take **different routes** through the network and may arrive **out of order**; the destination reassembles them.

Characteristics:

- **Efficient** — the same physical links can carry packets for many conversations simultaneously, statistically multiplexed.
- **Robust** — if a router or link fails, packets can be routed around the failure.
- **Variable latency** — different routes and queueing produce different delays.
- **Possible loss** — packets may be dropped if a router is congested. Reliability is the responsibility of higher layers (TCP).
- **Out-of-order arrival** — the destination must reassemble.

Used by: the **internet** (TCP/IP), modern cellular voice (VoIP), most modern networks.

### Comparing them

| Aspect | Circuit switching | Packet switching |
|---|---|---|
| Path | dedicated, reserved | shared, per-packet |
| Setup time | slow | none |
| Bandwidth use | inefficient | efficient |
| Order | in order | may be out of order |
| Robustness | a single failure cuts the circuit | reroute around failures |
| Suits | constant-rate continuous flows (traditional voice) | bursty, variable-rate flows (web traffic, email, video) |

Modern networks overwhelmingly use **packet switching** because of its efficiency and resilience. Circuit switching survives in some specialised legacy systems but the principle is increasingly emulated on top of packet networks where needed (e.g. "leased lines" provisioned over MPLS).

### Describing packet switching in a few sentences

A common exam answer: "The message is broken into small packets. Each packet has the destination address (and the source address, and a sequence number). Each packet is sent through the network independently, with routers making per-packet decisions about which next hop to forward it to. Packets may take different paths and may arrive out of order. The destination uses the sequence numbers to reassemble the original message; missing packets can be requested again."
