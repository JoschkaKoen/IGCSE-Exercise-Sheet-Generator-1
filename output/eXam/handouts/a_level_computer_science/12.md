## Program development life cycle

A **program development life cycle** is the sequence of stages from initial idea to finished, maintained software. It exists to **plan, manage and control** software projects — to make sure the right product is built, on time, with acceptable quality.

### Why a life cycle is needed

- **Manage complexity** — large programs cannot be written in one go; break into manageable phases.
- **Coordinate teams** — multiple people need an agreed sequence of work.
- **Track progress** — clear milestones show how much is done.
- **Ensure quality** — testing and review are planned in, not afterthoughts.
- **Document changes** — design decisions are recorded for future maintenance.
- **Manage risk** — problems are detected and addressed in the right stages.

### Why different life cycles exist

No single life cycle fits every project. The right choice depends on:

- the **size and complexity** of the project,
- how clear the **requirements** are at the start,
- how much **change** is expected during development,
- the **risk profile** of the project (safety-critical vs casual use),
- the **team size** and experience,
- the **deadline** and budget.

### Common life-cycle models

#### Waterfall

A **linear, sequential** model. Each stage (Analysis → Design → Coding → Testing → Maintenance) is **completed** before the next begins; there is little going back.

- **Benefits**: clear structure, easy to manage, well-documented at each stage. Suits projects with **stable, well-understood requirements**.
- **Drawbacks**: poor at coping with changes mid-project. Mistakes discovered late are expensive to fix. The customer doesn't see a working product until the end.

#### Iterative

Develop in **repeated passes** — each iteration produces a partially-complete version that is reviewed and refined before the next pass adds more.

- **Benefits**: feedback loop catches problems earlier. Suits projects where the **full requirements aren't known up front** but can be discovered.
- **Drawbacks**: harder to estimate total time and cost; risk of "rework" if early decisions need revisiting.

#### Rapid Application Development (RAD)

Heavy use of **prototyping** and end-user involvement. Develop quickly using existing components, get user feedback, refine.

- **Benefits**: very fast initial delivery; users get something to react to. Suits projects with **changing requirements** and where rapid time-to-market matters.
- **Drawbacks**: depends on user availability for feedback; less suitable for large, safety-critical systems; can produce systems with limited reusability.

#### Agile

A family of models (Scrum, Kanban, XP) using **short iterations** ("sprints"), continuous collaboration, frequent delivery and constant testing.

- **Benefits**: flexibility, fast adaptation, regular customer feedback.
- **Drawbacks**: requires committed customer and well-trained team; less appropriate when full requirements are needed up front for contractual reasons.

### The standard stages

Whatever the life cycle, most projects pass through these activities:

#### Analysis

Find out **what the program needs to do**. Talk to stakeholders, study the existing system, gather requirements, document them clearly. Produce a requirements specification.

#### Design

Decide **how** the program will meet the requirements. Choose data structures, algorithms, modules, user interface, file/database layouts. Produce design documents — flowcharts, structure charts, pseudocode outlines, ERDs, screen mock-ups.

#### Coding (implementation)

Write the actual source code in the chosen language, following the design. Includes integrating with libraries and any existing systems.

#### Testing

Run the program against test data and check the results. Detect, locate and fix bugs. Use the test strategy/plan (see below).

#### Maintenance

After release, keep the program working and useful. Fix any bugs discovered in use; adapt to changes in the environment; improve features.

## Program design tools

### Structure chart

A **structure chart** shows the **hierarchical decomposition** of a program into modules (subroutines), and the **parameters** passed between them.

- Each module is drawn as a rectangle, labelled with its name.
- Modules are connected by lines from caller to callee — caller above, callee below.
- Small arrows next to each line show the data passing in each direction:
  - small arrow with empty circle (data couple) — a simple data value passes,
  - small arrow with filled circle (flag / control couple) — a control flag passes.

Example sketch for a "Calculate Pay" program:

```
                CalculatePay
            /        |         \
       GetEmployee  CalculateBonus  CalculateTax
       Returns:     Takes: sales    Takes: gross
       employeeID   Returns: bonus  Returns: tax
```

(Arrows on the lines show the parameters going down and the return values going back up.)

A structure chart is a high-level **design tool**, used during the **design** stage. From it you can also derive the procedure signatures.

### State-transition diagram

A **state-transition diagram** shows the different **states** a system can be in and the **events** that cause it to move from one state to another. Used for systems with clear modes of operation — vending machines, traffic lights, parsers, user interfaces.

- Each state is drawn as a circle (or rounded rectangle), labelled.
- Each transition is drawn as an arrow from one state to another, labelled with the **event** that triggers it (and optionally an action that occurs during the transition).

Example: a vending machine.

```
   coin inserted               item selected
[Idle] ─────────────────→ [Awaiting selection] ───────────→ [Dispensing]
   ↑                                                              │
   │                                                              │ dispense complete
   └──────────────────────────────────────────────────────────────┘
```

State-transition diagrams **document an algorithm** clearly when its core logic is "be in state X until event Y occurs, then move to state Z". They make it easy to spot missing transitions ("what if the user inserts a second coin while awaiting selection?").

## Errors

Three kinds of program error to know.

### Syntax error

A violation of the **language's grammar rules**. The compiler or interpreter cannot produce a runnable program until they are fixed.

Examples: missing semicolon, misspelled keyword, unmatched bracket, `IF` without `ENDIF`.

Detected at compile / translation time. Fixed by reading the error message and correcting the offending line.

### Run-time error

An error that **occurs while the program is running**. The code compiles fine, but during execution something goes wrong.

Examples: dividing by zero, opening a file that doesn't exist, accessing an array index out of range, exceeding the stack depth from runaway recursion.

Run-time errors usually crash the program or trigger an exception. Detected by running the program; fixed by adding checks (range, file existence, sanitised input) or correcting the underlying logic.

### Logic error

The program runs **without crashing** but produces **wrong results**. The grammar and the runtime are fine; the algorithm is wrong.

Examples: using `+` where `-` was meant; off-by-one error in a loop; wrong order of conditions in an `IF...ELSE IF...` chain (e.g. checking `mark >= 40` before `mark >= 60`, so high marks get classified as "C").

Logic errors are usually the hardest to find — the only sign is that the output is wrong. Found by careful testing and by tracing the program with known inputs.

## Testing methods

### Dry run

Working through the code **on paper**, without running it on the computer. The tester traces each variable's value through every step, writing them down in a table. Useful for checking small algorithms before coding.

### Walkthrough

Like a dry run, but performed as a **team review** — one person walks through the code while others ask questions. Catches misunderstandings and design flaws.

### White-box testing

Testing that **looks at the internal structure** of the code. The tester designs test cases to cover every:

- statement (each line is run at least once),
- branch (each side of every `IF`/`ELSE` is taken at least once),
- loop (zero, one, multiple iterations).

White-box testing requires the tester to read the code.

### Black-box testing

Testing **based on the specification**, with no knowledge of the internals. The tester treats the program as a black box: feed inputs, observe outputs, check against expectations.

Black-box testing focuses on **what** the program should do — useful for users and independent testers.

### Integration testing

Once individual modules pass their tests, **combine them** and test the combined system. Catches errors at the **interfaces** between modules (mismatched parameter types, wrong order of arguments, unexpected sequencing).

### Alpha testing

Testing performed **by the developers or in-house testers** before the program is released to external users. Done in the developer's environment.

### Beta testing

Release the program to a **limited group of real end users** in their real environment. They use it, report bugs, suggest improvements. Helps find issues that don't show up in the lab.

### Acceptance testing

Final testing performed **by the customer** to decide whether the program is fit for purpose. The customer signs off when satisfied. The pass criterion is the requirements specification.

### Stub testing

A **stub** is a placeholder for a module that doesn't yet exist. While developing top-down, you can write the higher-level modules and use stubs for the lower-level ones (each stub just returns a fixed value or does nothing). Lets you test the structure of the program before all modules are written.

## Test strategy and test plan

A **test strategy** is the **high-level approach** to testing — what kinds of testing will be done, who does them, in what order, with what tools. Typical contents:

- which testing methods to use (white-box, black-box, alpha, beta),
- who is responsible for each kind of testing,
- when each kind of testing happens in the schedule,
- the acceptance criteria for moving from one stage to the next.

A **test plan** is the **detailed plan for actually running the tests**. Typical contents:

- the items to be tested (subroutine, screen, full system),
- the test cases — each with input data, expected output, and an actual-output column to fill in,
- the test environment (hardware, OS, data sets),
- pass/fail criteria for each test.

### Choosing test data

For each test, pick data that **exposes possible faults**. Three categories:

- **Normal data** — values within the expected range of normal use. Tests that the program handles the typical case correctly. For an exam-mark field 0–100, normal data might be `50`, `75`.
- **Abnormal data** (also called **erroneous** or **invalid**) — values that should be rejected. Tests that the validation works. For the same field, abnormal data: `-10`, `200`, `"abc"`.
- **Extreme / boundary data** — values at the **edges** of the valid range. Off-by-one errors live here. For 0–100, boundary data: `0`, `100` (just inside the range), and sometimes `-1`, `101` (just outside).

For each new field or condition the program checks, include a normal, an abnormal, and a boundary case in the test plan.

## Maintenance

Once a program is released, it **must continue to be maintained**. Most of a program's lifetime cost is in maintenance, not initial development. Three kinds:

### Perfective maintenance

Improving the program's performance, efficiency or features even though it is working correctly. Examples: rewriting a slow query to be faster; restructuring code for better readability; adding a more convenient option.

Driven by a **desire to improve** the existing system.

### Adaptive maintenance

Changing the program so it **continues to work** in a changing environment. Examples: porting to a new operating system; updating to use a newer API; supporting a new database engine; adapting to legal/regulatory changes.

Driven by **changes outside the program**.

### Corrective maintenance

**Fixing bugs** — defects that escaped testing and were discovered in use. Examples: a calculation that produces the wrong answer for certain inputs; a screen that doesn't refresh correctly; a crash on specific data.

Driven by **fault reports** from users or monitoring.

A real program may need all three kinds throughout its life — sometimes simultaneously.

## Amending an existing program

You may be given an existing program and asked to **add a feature or fix a bug**. Steps:

1. **Read the existing code carefully** until you understand the algorithm and the data flow.
2. **Identify where the change goes** — which subroutine, which lines.
3. **Make the change** as small as possible — don't rewrite working code.
4. **Update related parts** — if you change a parameter list, update every caller. If you change a data structure, update every routine that uses it.
5. **Test** — both the new behaviour and the existing behaviour (regression testing — make sure you didn't break anything).
6. **Document** the change.

A well-written program with **clear comments**, **meaningful identifier names**, **decomposed subroutines** and **a structure chart** is much easier to amend than a monolithic one. That's why the design tools earlier in this topic matter even after the initial development.
