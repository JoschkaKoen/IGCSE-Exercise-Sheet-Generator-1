## File-based storage and its limitations

Before databases became common, programs stored data in **flat files** — typically one file per program. This works for small amounts of data but breaks down in larger applications.

### Limitations

- **Data redundancy** — the same data (e.g. a customer's address) is stored in multiple files for different programs. Wastes space and makes errors more likely.
- **Data inconsistency** — when redundant copies of the same data are updated separately, they get out of sync. A customer's new address might be in the orders file but not in the marketing file.
- **Data dependence** — programs are tightly coupled to the file format. If the file structure changes, every program that reads it must be rewritten.
- **Difficult to enforce data integrity** — each program is responsible for validating its own data; rules can be inconsistent or forgotten.
- **Difficult to share** — concurrent access by multiple programs requires careful file locking, which is rarely done well.
- **Limited querying** — extracting data combining several files (e.g. "all customers whose last order was over £100") requires custom code.
- **Security** — files can be read by anyone with file-system access; fine-grained per-user permissions on specific fields are difficult.

A **relational database** addresses these limitations by storing data in **tables** managed by a single piece of software (the DBMS) that all programs use to access the data.

## Relational database model — terminology

A **relational database** organises data into **tables** with the following structure.

- **Table** (or **relation**) — a two-dimensional grid of rows and columns. Each table represents one type of entity (e.g. `CUSTOMER`, `ORDER`).
- **Record** (or **row, tuple**) — one row of the table. Represents one instance of the entity (e.g. one customer).
- **Field** (or **column, attribute**) — one column of the table. Represents one piece of information about every record (e.g. `Phone`).
- **Primary key** — a field (or combination of fields) whose value **uniquely identifies** each record in the table. Cannot be null, cannot duplicate. Conventionally underlined when writing table definitions. Examples: `CustomerID`, `OrderNumber`.
- **Foreign key** — a field in one table whose value **matches the primary key of another table** — used to link the two tables. E.g. `CustomerID` in the `ORDER` table is a foreign key referencing `CUSTOMER.CustomerID`.
- **Composite key** — a primary key made up of two or more fields together (no single field is unique on its own).
- **Candidate key** — any field (or combination) that could serve as a primary key — one is chosen, the others become alternates.
- **Referential integrity** — every foreign-key value must match an existing primary-key value in the referenced table (no "orphan" records).

A table is often written in shorthand as `TABLE(field1, field2, ..., fieldN)` with the primary key underlined and foreign keys identified separately, e.g.:

```
CUSTOMER(CustomerID, Name, Phone)
ORDER(OrderID, CustomerID, OrderDate)   -- CustomerID is FK → CUSTOMER
```

## Entity-relationship (E-R) diagrams

An **E-R diagram** documents the structure of a database visually:

- Each **entity type** (table) is drawn as a rectangle, labelled with its name.
- Each **relationship** between two entities is drawn as a line between them.
- The **cardinality** of a relationship is marked at each end:
  - **one-to-one (1:1)** — each record on one side matches at most one on the other (e.g. each Employee has one PayrollRecord).
  - **one-to-many (1:M)** — each record on one side may match many on the other (e.g. each Customer can place many Orders; each Order has one Customer).
  - **many-to-many (M:N)** — each record on either side may match many on the other (e.g. Students enrol on many Courses; each Course has many Students).

A many-to-many relationship cannot be implemented directly in a relational database. It must be broken into two one-to-many relationships through a **link table** (junction table) that holds the foreign keys of the two related tables. For Student–Course:

```
ENROLMENT(StudentID, CourseID, EnrolmentDate)
```

(`StudentID` and `CourseID` together form the composite primary key; each is also a foreign key.)

## Normalisation

**Normalisation** is the process of organising tables to **minimise redundancy and inconsistency**. You proceed through normal forms in order.

### First Normal Form (1NF)

- Every field contains a **single value** (atomic).
- No **repeating groups** of fields.
- Each row is uniquely identifiable (has a primary key).

A table where one column holds a list of items (e.g. `Subjects = "Maths, Physics"`) violates 1NF — split into separate rows or another table.

### Second Normal Form (2NF)

- Already in 1NF.
- Every non-key field depends on the **whole** of the primary key, not just part of it. (Only relevant if the primary key is **composite**.)

If a composite key is `(OrderID, ProductID)` and the table has `ProductName` (which depends only on `ProductID`, not on the whole key), it violates 2NF. Move `ProductName` to a separate `PRODUCT` table.

### Third Normal Form (3NF)

- Already in 2NF.
- Every non-key field depends **only** on the primary key — not on another non-key field (no **transitive dependency**).

If a `CUSTOMER` table has `(CustomerID, Name, City, PostalCode, Country)` where `Country` depends on `City` (which depends on `CustomerID`), then `Country` transitively depends on `CustomerID` through `City`. Move `(City, Country)` to a separate table.

### Why 3NF is desirable

A 3NF schema:

- has **no redundancy** of facts — each fact stored once.
- **insert, update, delete anomalies disappear** — updating a fact in one place updates it everywhere.
- is easier to maintain consistency under change.

Trade-off: more tables = more joins to assemble data for a query. Sometimes performance demands controlled de-normalisation. At A-Level, aim for 3NF unless told otherwise.

### Producing a 3NF design

Given a flat starting table or a description:

1. Identify the entities (likely tables) and their natural attributes.
2. Choose a primary key for each.
3. Check 1NF: split repeating groups / atomic-violating fields.
4. Check 2NF: for any composite key, split fields that depend on only part of it.
5. Check 3NF: split fields that depend transitively (via another non-key field) on the primary key.
6. Add foreign keys to express the relationships between tables.

## Database Management System (DBMS)

A **DBMS** is software that manages a database — providing centralised access, integrity, security and queries. Examples: MySQL, PostgreSQL, Oracle, Microsoft SQL Server, SQLite.

### Features that address file-based limitations

- **Data dictionary / catalogue** — a description of every table, field, type, key, and relationship. Programs query this rather than hard-coding the structure, so changes propagate.
- **Data redundancy and consistency control** — each fact stored once; updates apply once and are seen by all programs.
- **Concurrent access control** — locks, transactions and isolation levels let many users read and write simultaneously without corrupting each other's work.
- **Backup and recovery** — automatic backups; the DBMS can roll back incomplete transactions after a crash.
- **Security and access control** — per-user / per-role permissions on tables, views and even fields. Different users see different things.
- **Data integrity rules** — primary keys, foreign keys, unique constraints, range / format constraints. The DBMS enforces them centrally; no program can write data that violates them.
- **Query language** — a powerful language (SQL) for retrieving and updating data.
- **Views** — virtual tables defined by queries; users see "their" view of the data, hiding the underlying structure.
- **Transactions** — group of operations that succeed or fail as a unit.

### Software tools provided by a DBMS

- **Data dictionary tool** — view and edit table and field definitions.
- **Query builder** — construct SQL queries graphically (drag in tables, pick fields, set conditions). Useful for non-programmers.
- **Forms builder** — build data-entry forms that let end users add/edit records without writing SQL.
- **Report generator** — produce formatted printable / on-screen reports based on queries.
- **User-and-permission management** — set up accounts, roles and access rights.
- **Backup and restore tools**.
- **SQL editor / shell** — run ad-hoc SQL.

## DDL and DML

SQL — **Structured Query Language** — is the industry standard for both creating database structures and working with the data. SQL has two halves:

- **Data Definition Language (DDL)** — statements that **create or modify the structure** of the database (the schema): tables, fields, keys, constraints.
- **Data Manipulation Language (DML)** — statements that **work with the data** inside the structure: insert, update, delete, query.

### DDL — basics

You should recognise and be able to write these simple DDL statements.

**Create a table**:

```sql
CREATE TABLE CUSTOMER (
  CustomerID INTEGER PRIMARY KEY,
  Name VARCHAR(50) NOT NULL,
  Phone VARCHAR(20)
);
```

The `CREATE TABLE` statement names the table and lists each field with its type and any constraints (`PRIMARY KEY`, `NOT NULL`, `UNIQUE`).

**Add a foreign key**:

```sql
CREATE TABLE ORDER (
  OrderID INTEGER PRIMARY KEY,
  CustomerID INTEGER,
  OrderDate DATE,
  FOREIGN KEY (CustomerID) REFERENCES CUSTOMER(CustomerID)
);
```

**Modify a table**:

```sql
ALTER TABLE CUSTOMER ADD Email VARCHAR(100);
```

**Drop a table**:

```sql
DROP TABLE CUSTOMER;
```

Common data types: `INTEGER`, `VARCHAR(n)` (string up to $n$ characters), `CHAR(n)` (fixed length), `DATE`, `BOOLEAN`, `DECIMAL(p, s)`.

### DML — basics

**Query (SELECT)** — retrieve data:

```sql
SELECT Name, Phone
FROM CUSTOMER
WHERE City = 'London'
ORDER BY Name ASC;
```

`SELECT` lists the fields to return; `FROM` names the table(s); `WHERE` filters rows; `ORDER BY` sorts (`ASC` or `DESC`).

**Joining two tables** — combine data using a foreign-key relationship:

```sql
SELECT CUSTOMER.Name, ORDER.OrderDate
FROM CUSTOMER, ORDER
WHERE CUSTOMER.CustomerID = ORDER.CustomerID
  AND ORDER.OrderDate >= '2024-01-01';
```

This returns the name and order date for every customer order placed since the start of 2024. The `WHERE` clause's first condition is the **join condition** — pairs each order with the customer whose `CustomerID` matches.

A modern alternative syntax (also fine in the exam):

```sql
SELECT C.Name, O.OrderDate
FROM CUSTOMER C INNER JOIN ORDER O
  ON C.CustomerID = O.CustomerID
WHERE O.OrderDate >= '2024-01-01';
```

**Aggregate functions**: `COUNT(*)`, `SUM(field)`, `AVG(field)`, `MIN(field)`, `MAX(field)`. Often used with `GROUP BY`:

```sql
SELECT CustomerID, COUNT(*) AS NumOrders
FROM ORDER
GROUP BY CustomerID;
```

Returns the number of orders for each customer.

**Insert** a new row:

```sql
INSERT INTO CUSTOMER (CustomerID, Name, Phone)
VALUES (101, 'Ada Lovelace', '020-1234-5678');
```

**Update** an existing row:

```sql
UPDATE CUSTOMER
SET Phone = '020-9999-0000'
WHERE CustomerID = 101;
```

**Delete** a row:

```sql
DELETE FROM CUSTOMER
WHERE CustomerID = 101;
```

Always include a `WHERE` clause on `UPDATE` and `DELETE` — without it, the operation affects **every** row.

### Tips for writing exam SQL

- Use the exact table and field names from the schema given in the question (case may matter).
- Quote string literals with single quotes (`'Smith'`); don't quote numbers.
- Use `=`, `<`, `>`, `<=`, `>=`, `<>` (or `!=`) for comparisons.
- `LIKE 'A%'` matches anything starting with A; `%` wildcards any string; `_` wildcards one character.
- `IN (1, 2, 3)` matches any value in the list.
- `BETWEEN 10 AND 20` matches values from 10 to 20 inclusive.
- Use `AND` / `OR` / `NOT` to combine conditions.
- End each statement with a semicolon.
