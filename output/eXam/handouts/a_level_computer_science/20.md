## Programming paradigms

A **programming paradigm** is a style or model of programming — a way of structuring programs that comes with its own ideas, vocabulary and supporting language features.

The four paradigms in this syllabus.

### Low-level programming

Programming **close to the hardware** in **machine code** or **assembly language**. Each instruction maps directly to what the CPU executes.

Characteristics:

- Direct access to **registers**, **memory addresses**, **I/O ports**.
- Architecture-specific (one CPU family).
- Very fast and memory-efficient — every instruction is under the programmer's control.
- Tedious — long programs even for simple tasks.
- Hard to maintain — little abstraction.

Used for: device drivers, embedded firmware, performance-critical inner loops, bootloaders.

### Imperative (procedural) programming

The programmer writes a **sequence of commands** that change the program's state — assignments, conditionals, loops, function calls. The dominant style of the languages most programmers learn first (Python, C, Pascal, BASIC).

Characteristics:

- **Variables** hold state; **statements** modify it.
- Code is organised into **procedures and functions** ("structured programming").
- Control flow uses **sequence, selection, iteration** explicitly.
- Same as the algorithm-design style covered in Topics 9 and 11.

Strong when: the algorithm has clear sequential steps and well-defined state.

### Object-oriented programming (OOP)

Programs are built from **objects** — self-contained units that combine **data** (attributes) and **operations on that data** (methods). Objects are instances of **classes**, which define the shape and behaviour.

Characteristics (the four pillars):

- **Encapsulation** — an object's data is hidden behind its methods. External code can only interact with the object through its public methods, not by reading or writing the data directly. Protects the object's invariants and lets the internal representation change without breaking callers. Implemented with **public / private / protected** access modifiers.
- **Inheritance** — a class (the **subclass** or **derived class**) can be defined as a specialisation of another (the **superclass** or **base class**). The subclass inherits all the attributes and methods of the superclass, and can add its own or override the inherited ones. Promotes reuse and models "is-a" relationships ("a Manager is an Employee").
- **Polymorphism** — different objects can respond to the **same method call** with different behaviour. The caller doesn't need to know the exact type of the object. E.g. every `Shape` has `Area()`; a `Circle` and a `Rectangle` each implement it differently; code that asks any `Shape` for its area works for both.
- **Abstraction** — present a simplified interface to users of the class while hiding implementation detail. Closely related to encapsulation.

Other OOP terminology:

- **Instance** — a concrete object created from a class (e.g. `myCar` is an instance of class `Car`).
- **Constructor** — a special method called when a new object is created; initialises the attributes.
- **Method** — a function defined inside a class, operating on the object's data.
- **Attribute (field, property)** — a variable defined inside a class, holding an object's data.

Used for: large systems, GUIs, simulations, games — anywhere modelling real-world entities helps structure the code.

### Declarative programming

The programmer specifies **what** the program should compute, not **how** to compute it step by step. The language's runtime works out the steps.

Two sub-paradigms:

- **Functional programming** — programs are built from **pure functions** (no side effects; same input always produces same output) composed together. Examples: Haskell, Lisp; functional features in modern Python, JavaScript, Java.
- **Logic programming** — the programmer states facts and rules; the engine answers queries by inference. Example: Prolog.

A familiar declarative example: **SQL**. You write `SELECT * FROM Customer WHERE Country = 'UK' ORDER BY Name` — you say what you want, not how to walk through pages of records.

Used for: data queries (SQL), AI knowledge systems (Prolog), maths-heavy code where avoiding mutation makes reasoning easier, configuration languages.

### Comparing paradigms

| Paradigm | Strength | Typical languages |
|---|---|---|
| Low-level | maximum control, speed | assembly, machine code |
| Imperative / procedural | direct, intuitive | C, Python, Pascal |
| Object-oriented | modular, models entities | Java, C#, Python, C++ |
| Declarative (functional) | clear, no side effects | Haskell, F#, Scala |
| Declarative (logic) | inference, rules | Prolog |
| Declarative (database) | data queries | SQL |

Modern languages often **mix paradigms** — Python supports procedural, OOP and functional features. The right paradigm depends on the problem.

## File processing

This topic extends the file handling from Topic 10 with code-level operations. Pseudocode operations:

- **OPENFILE** — `OPENFILE "data.txt" FOR READ | WRITE | APPEND`. `READ` opens an existing file for reading; `WRITE` creates a new file (overwriting any existing); `APPEND` opens for writing at the end of an existing file.
- **READFILE** — `READFILE "data.txt", line` reads the next line into `line`. Use a `WHILE NOT EOF(...)` loop to read to the end.
- **WRITEFILE** — `WRITEFILE "data.txt", "value"` writes one record / line.
- **CLOSEFILE** — `CLOSEFILE "data.txt"`. Always close.
- **EOF** — `EOF("data.txt")` returns TRUE when there is no more data to read.

### Reading a whole file

```
OPENFILE "names.txt" FOR READ
WHILE NOT EOF("names.txt") DO
    READFILE "names.txt", thisName
    OUTPUT thisName
ENDWHILE
CLOSEFILE "names.txt"
```

### Writing a file

```
OPENFILE "log.txt" FOR WRITE
FOR i ← 1 TO 10
    WRITEFILE "log.txt", "Event " & i
NEXT i
CLOSEFILE "log.txt"
```

### Appending

```
OPENFILE "log.txt" FOR APPEND
WRITEFILE "log.txt", "New event at end"
CLOSEFILE "log.txt"
```

### Searching a file

```
target ← "Smith"
found ← FALSE
OPENFILE "people.txt" FOR READ
WHILE NOT EOF("people.txt") AND NOT found DO
    READFILE "people.txt", line
    IF line = target THEN
        found ← TRUE
    ENDIF
ENDWHILE
CLOSEFILE "people.txt"

IF found THEN
    OUTPUT "Found"
ELSE
    OUTPUT "Not found"
ENDIF
```

### Updating a file in place

Most languages don't let you update a text file in place. Instead:

1. Open the original for **READ**, and a temporary file for **WRITE**.
2. For each line of the original:
   - if it should change → write the new version to the temp file,
   - else → write the original line unchanged.
3. Close both files.
4. Replace the original with the temp file.

This pattern works for deleting lines (skip them) and inserting lines (write extra) too.

### Pitfalls

- **Forgetting to close** a file — data may be lost.
- **Opening for WRITE** when you meant **APPEND** — overwrites everything.
- **Reading past EOF** — check `EOF` before reading.
- **Hard-coded paths** — make file paths into parameters / constants so the program is portable.

## Exception handling

An **exception** is an **error or unexpected condition** that arises during program execution — e.g. division by zero, file not found, network failure, invalid user input, an array index out of range.

**Exception handling** is the language feature that lets a program **detect** an exception and **respond** to it gracefully, instead of crashing.

### Why exception handling matters

- Real-world programs face many error conditions that **cannot be prevented up-front** — files may be moved, networks may fail, users may type the wrong thing.
- Without exception handling, every operation must be checked separately with `IF`s, cluttering the code.
- Crashing on every error is unacceptable in any serious software.
- Exception handling **separates the normal flow** from the error-handling logic — the "happy path" reads cleanly and the error responses are in dedicated blocks.

### Pseudocode pattern

```
TRY
    OPENFILE "data.txt" FOR READ
    READFILE "data.txt", line
    OUTPUT line
    CLOSEFILE "data.txt"
EXCEPT FileNotFound
    OUTPUT "Sorry, the file does not exist."
EXCEPT ReadError
    OUTPUT "Sorry, error reading the file."
ENDTRY
```

- The `TRY` block contains the code that might fail.
- One or more `EXCEPT` blocks handle specific kinds of exception. The first matching block runs; the rest are skipped.
- After the `EXCEPT` (or after the `TRY` if no exception occurred), execution continues with the code that follows.

Most real languages also support:

- a **catch-all** `EXCEPT` that matches any exception not caught above,
- a `FINALLY` clause that runs **whether or not** an exception occurred — useful for cleanup (close files, release resources).

### Raising / throwing an exception

A subroutine that detects an error can **raise** an exception itself, so the caller can handle it:

```
PROCEDURE Divide(a : INTEGER, b : INTEGER) RETURNS INTEGER
    IF b = 0 THEN
        RAISE DivideByZero
    ENDIF
    RETURN a DIV b
ENDPROCEDURE
```

The caller can wrap the call in a `TRY...EXCEPT DivideByZero` and respond appropriately.

### Where to handle exceptions

- **Close to where the error happens** if the action is simple (display a message, retry, fall back).
- **Higher up the call stack** if only the outer code knows what to do (e.g. a top-level loop in a GUI app catches all exceptions, logs them, shows the user a friendly error dialog, and keeps the app running).
- **Don't swallow exceptions silently** — at minimum, log them. Hiding errors makes debugging impossible.

### Common exceptions to handle

- `FileNotFound` — file does not exist.
- `IOError` — generic I/O failure.
- `DivisionByZero` — denominator is zero.
- `IndexOutOfRange` — array index outside the bounds.
- `InvalidArgument` — input value doesn't meet a precondition.
- `NullReference` — operating on an unassigned pointer / reference.
- `OutOfMemory` — allocation failed.

Wrap each potentially-failing operation in a `TRY` block with appropriate `EXCEPT` handlers. The result is a program that **degrades gracefully** under unexpected conditions instead of crashing — essential in any production code.
