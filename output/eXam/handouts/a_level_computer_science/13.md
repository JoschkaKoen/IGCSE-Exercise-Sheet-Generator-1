## User-defined data types

The built-in types (`INTEGER`, `REAL`, `STRING`, `CHAR`, `BOOLEAN`) cover the simplest cases. For richer problems you can **define your own types** — making the program clearer and the compiler more helpful.

### Why user-defined types are needed

- The built-in types don't capture **domain meaning**. A vehicle identifier like `"M100"` is *not* just any string — it's one of a small fixed set. A built-in `STRING` lets you assign nonsense values; a user-defined type can restrict to legal values.
- A real-world entity is usually a **collection** of values of different types. Built-in types are too low-level to model a customer, an order or a date by themselves.
- Code becomes **self-documenting**: `DECLARE Taxi : Vehicle` is clearer than `DECLARE Taxi : STRING`.
- The compiler / interpreter can **type-check** more strictly, catching errors at translation time.

### Non-composite user-defined types

A **non-composite** type still represents a single value, but one drawn from a programmer-defined set.

#### Enumerated type

Defines a type whose values are a **fixed list of named constants**. Use when a variable can be only one of a few options.

```
TYPE Vehicle = (M100, M230, T101, T102, T120, T150)

DECLARE MyTaxi : Vehicle
MyTaxi ← T102
```

The names are not strings — they are values of the new type. Internally the language usually stores them as small integers. You cannot accidentally assign a value that isn't in the list.

Common uses: days of the week, colours, status codes, vehicle types, suit of a playing card.

#### Pointer type

A **pointer** holds the **memory address of another variable** (or `NULL` / `NIL` to indicate "no target"). Pointers let you build dynamic data structures (linked lists, trees) and pass references to large objects without copying them.

```
TYPE PNode = ^TNode    // pointer to a TNode

DECLARE p : PNode
p ← NEW TNode          // allocate a new node, store its address in p
p^.Value ← 42          // dereference to access fields
```

Operations on pointers: **dereference** (`p^`) to access the variable pointed to; **assign** another pointer; compare to `NULL`. **Dispose** when no longer needed, to free memory.

### Composite (structured) user-defined types

A **composite** type bundles **several values** under one identifier.

#### Record

Already covered (Topic 10). A `TYPE ... RECORD ... ENDTYPE` block groups fields of different types.

```
TYPE TBooking
    DECLARE BookingID : INTEGER
    DECLARE Pickup : STRING
    DECLARE Dropoff : STRING
    DECLARE Taxi : Vehicle
    DECLARE Time : DATE
ENDTYPE
```

#### Set

A **set** is an unordered collection of unique values from some base type. Common operations: add, remove, test membership, union, intersection, difference.

```
TYPE Colour = (Red, Green, Blue, Yellow)
DECLARE Available : SET OF Colour
Available ← {Red, Blue}
IF Green IN Available THEN ...
```

Sets are useful when the order doesn't matter and you mostly ask "does X belong?".

#### Class / object

A **class** is the OOP composite type — combines data fields (attributes) with operations on those data (methods). An **object** is an instance of a class.

```
CLASS Taxi
    PRIVATE Identity : Vehicle
    PRIVATE Capacity : INTEGER

    PUBLIC PROCEDURE NEW(id : Vehicle, cap : INTEGER)
        Identity ← id
        Capacity ← cap
    ENDPROCEDURE

    PUBLIC FUNCTION GetCapacity() RETURNS INTEGER
        RETURN Capacity
    ENDFUNCTION
ENDCLASS

DECLARE MyTaxi : Taxi
MyTaxi ← NEW Taxi(T102, 4)
```

A class is the foundation for the OOP paradigm — covered more in Topic 20 (Further Programming).

### Choosing the right user-defined type

When designing a program, list the kinds of data and pick:

- **enumerated** for a value drawn from a fixed list of named options,
- **pointer** when you need indirection (dynamic structures, modifying via reference),
- **record** for a group of related fields,
- **set** for an unordered, unique collection,
- **class** when you need both state **and** behaviour together, or polymorphism / inheritance.

For a taxi-booking system you'd typically define `Vehicle` (enumerated), `TBooking` (record), `Customer` (record or class), and use arrays or sets to hold collections of them.

## File organisation and access

A program often needs to store data **on disk**. The **organisation** is how the data is laid out in the file; the **access method** is how the program reaches a particular record. The right choice depends on how the data is used.

### Serial file

Records stored in the **order they were added**. No sorting, no indexing.

- **Access**: only **sequential** — start at the front, walk through until you find what you want.
- **Insert**: append at the end — fast.
- **Search**: slow on average — half the file.
- **Use cases**: transaction logs, audit trails, anything append-only.

### Sequential file

Records stored **sorted by a key field**.

- **Access**: usually **sequential**, but the sort lets you stop early when the key you want would have appeared, or use a binary search (if the medium supports jumping). Direct access by key is possible if the file structure supports it.
- **Insert**: slow — to maintain order, you must find the right place and shift subsequent records.
- **Search**: faster than serial because the sort order lets you stop or binary-search.
- **Use cases**: backup tapes, master files updated in batch (a typical batch run reads the master file in key order alongside a sorted transaction file).

### Random (direct-access) file

Records stored at **positions calculated from the record key**, often using a hash function.

- **Access**: **direct** — given a key, compute the address and go straight to the record.
- **Insert / search / update**: very fast for individual records (O(1) on average).
- **Sequential access**: harder, because the records are not in key order.
- **Use cases**: large lookup tables, customer accounts, anywhere fast key-based access is critical.

### Methods of file access

- **Sequential access**: read records from start to end in their physical order. Used by serial and sequential files.
- **Direct access**: jump to a record at a known position. Used by random files and (for some operations) sequential files.

### Choosing organisation and access

Consider:

- How often is the file **searched** — by which fields? (Search by primary key dominates → random; search by date order → sequential.)
- How often are records **added** vs **modified** vs **read**?
- How big is the file? (Random with hashing scales well; sequential becomes painful at large size.)
- Will the file be **read in order** (e.g. for a report) or by **single key lookups** (e.g. for a website)?

Match the structure to the dominant operation.

## Hashing algorithms

A **hash function** takes a record key (e.g. an integer, a string) and produces an **address** (an integer index into a file) where that record should be stored.

A good hash function:

- is **fast** to compute,
- **distributes keys evenly** across the address space (so few collisions),
- is **deterministic** — the same key always produces the same address.

### Simple hashing examples

For an address space of $N$ slots (positions $0$ to $N-1$):

- **Modulo hash**: `address ← key MOD N`. For numeric keys.
- **Folding**: split the key into equal-length pieces, add them together, take MOD N.
- **Mid-square**: square the key, take some middle digits.
- **String hash**: sum the character codes (or compute a polynomial in them), MOD N.

### Collisions

A **collision** occurs when two different keys hash to the **same address**.

Resolution strategies:

- **Linear probing**: if the slot is occupied, try the next one ($address + 1, address + 2, \ldots$ wrapping around). Simple but causes clustering.
- **Chaining**: each slot holds a pointer to a **linked list** of all records that hashed there. Search the list when looking up a record.
- **Rehashing**: apply a second hash function to find an alternative slot.

To **search** for a key in a hashed file:

1. Compute the hash address from the key.
2. Read the record at that address.
3. If the keys match, done.
4. If not (collision), follow the chosen resolution strategy (next slot, next link in the chain) until you find the matching record or reach an empty slot (meaning the record is not in the file).

To **insert**:

1. Compute the hash address.
2. If the slot is free, write the record there.
3. Otherwise, follow the collision-resolution strategy until you find a free slot.

A good design keeps the **load factor** (records / slots) below about 70% to keep average lookup time near O(1).

## Floating-point numbers

To represent **real numbers** with very different magnitudes (atomic scales to astronomical), computers use a **floating-point** format — a binary version of scientific notation.

### Format

A floating-point number is stored as two fields:

- a **mantissa** (sometimes called significand) — the significant digits,
- an **exponent** — the power of 2 by which the mantissa is multiplied.

Both fields are stored as **two's-complement** binary integers in this syllabus.

The value represented is:

$$
\text{number} = \text{mantissa} \times 2^{\text{exponent}}.
$$

The mantissa is typically a **fixed-point fraction**: think of an implied **binary point** just after the sign bit. So if the mantissa field is 8 bits `0.1010000` (binary point after the leftmost bit), it represents the binary fraction $0.1010000_{2} = 1/2 + 1/8 = 0.625$.

### Example

Let mantissa be 8 bits (with the binary point after bit 7, i.e. the leftmost bit is the sign / "ones" place) and exponent be 8 bits.

Mantissa = `0.1010000` (with leading sign bit treated as $\pm 1$), exponent = `00000010` (= 2).

Value = $0.625 \times 2^{2} = 2.5$.

### Converting binary floating-point to denary

1. Read the mantissa. If the sign bit (leftmost) is 1, the mantissa is negative — find the magnitude using two's-complement rules.
2. Convert the mantissa to its denary fractional value (treat the bits as a fraction with the binary point in the agreed position).
3. Read the exponent. If it's negative (two's complement), find its value as a signed integer.
4. Multiply the mantissa by $2^{\text{exponent}}$.

### Converting denary to binary floating-point

1. Convert the magnitude of the denary number to a binary fraction × power of 2 (this is normalised binary scientific notation).
2. Write the mantissa in the agreed fixed-point format.
3. Write the exponent in the agreed two's-complement format.
4. Apply the sign as needed.

Worked example. Convert $2.5_{10}$ to floating-point with 8-bit mantissa (binary point after bit 7) and 8-bit exponent.

- $2.5_{10} = 10.1_{2} = 0.101_{2} \times 2^{2}$ (normalised — see below).
- Mantissa = `0.1010000` (the bits after the implied binary point), with positive sign bit. In full 8-bit form `0.1010000`.
- Exponent = `00000010` (positive 2 in 8-bit two's complement).

### Normalisation

A number is **normalised** when its mantissa has the **first significant bit immediately after the binary point** — that is, the mantissa's most significant magnitude bit is 1 (for positive numbers) or 0 (for negative numbers in two's complement); equivalently, no leading zeros (or leading ones) waste space.

Normalisation **maximises the precision** of the representation, because every bit of the mantissa is meaningful — none is wasted on leading zeros.

To **normalise**:

- shift the mantissa left until the first significant bit is just after the binary point, and decrement the exponent each shift; or
- shift the mantissa right and increment the exponent (if it was over-large to start with).

Examples (8-bit positive mantissa):

- `0.0010100` exponent `00000101` (= 5) → shift mantissa left 2 → `0.1010000` exponent `00000011` (= 3). Value unchanged.
- `0.0001000` exponent `00000010` (= 2) → shift mantissa left 3 → `0.1000000` exponent `11111111` (= -1). Value unchanged.

For negative numbers (two's-complement mantissa), normalised form has the **sign bit** (1) followed immediately by a 0 — the first bit different from the sign bit comes right after the binary point.

### Approximation and rounding errors

Many denary real numbers **cannot be represented exactly** in binary floating-point — just as $1/3$ cannot be written exactly in finite denary. The classic example: $0.1_{10}$ in binary is the repeating fraction $0.0001100110011\ldots_{2}$, which must be truncated to fit the mantissa. So a computer storing $0.1$ actually stores something very close but not exactly equal.

Consequences:

- **Rounding errors**: small differences accumulate over many operations. `(0.1 + 0.2) = 0.30000000000000004` in many floating-point systems, not exactly `0.3`.
- **Comparisons fail**: `x = 0.3` may be false even when the algebra says it should be true. Compare with tolerance: `ABS(x - 0.3) < 1e-9`.
- **Loss of precision**: subtracting two nearly-equal floating-point numbers leaves few significant bits — "catastrophic cancellation".

For applications where exactness matters (currency, accounting), use **fixed-point** or **BCD** representations instead of binary floating-point. For scientific computation, design algorithms that minimise error accumulation, and use the largest precision available (typically 64-bit or higher).
