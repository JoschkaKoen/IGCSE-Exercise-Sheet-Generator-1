## Operating Systems

### Why a computer needs an OS

A computer's hardware on its own is just silicon — it can fetch and execute instructions but knows nothing about files, processes, networks or users. The **operating system (OS)** is the **software layer** that:

- **Manages the hardware** (processor, memory, I/O devices, storage) on behalf of running programs.
- **Provides services** (file system, network access, user accounts, time, etc.) through a well-defined interface so programs don't have to talk to the hardware directly.
- **Provides a user interface** (command line, GUI, touch) so a human can launch programs and manage files.
- **Allows multiple programs to share the hardware safely** — each gets a fair share of CPU time, memory protection from each other, controlled access to devices.

Without an OS each program would have to include its own drivers and resource-management code, the user would have to load programs from scratch, and only one program could safely run at a time.

### Key management tasks

A typical OS performs (at least) these jobs:

- **Process management** — load programs into memory, schedule them on the CPU, switch between them so it appears many are running simultaneously, kill processes that misbehave.
- **Memory management** — allocate memory to processes, keep them out of each other's memory space, use paging or virtual memory so that the working set of all running processes can exceed physical RAM.
- **File management** — organise files and directories on secondary storage, handle reading and writing, control permissions, prevent simultaneous-write corruption.
- **Device / peripheral management** — handle I/O via device drivers; buffer data; manage interrupts from devices; provide a uniform interface so applications don't need to know the details of each device.
- **Security management** — user accounts and passwords, file permissions, network firewall, antivirus integration, encryption.
- **User interface** — present a GUI or CLI for the user to interact with running programs and the OS itself.
- **Networking** — manage network connections, sockets, IP configuration.

### Utility software

Most OSes come with a set of **utility programs** — small programs that help users maintain or configure the system. Typical utilities:

- **File / disk management** — copy, move, delete files; format drives; check disk integrity (`chkdsk`).
- **Disk defragmenter** — reorganises fragmented files on a magnetic hard disk so that contiguous blocks are stored together, reducing seek times. (Not useful on SSDs.)
- **Backup** — copies user data to other storage, often automatically on a schedule, so it can be recovered after failure or accidental deletion.
- **Antivirus / antimalware** — scans files and memory for known malware signatures, monitors for suspicious behaviour, quarantines threats.
- **Firewall** — filters incoming and outgoing network traffic according to rules; blocks unsolicited inbound connections.
- **Compression / archiving** — creates compact archives of files; `zip`, `tar`, `gzip`.
- **System monitor / task manager** — lets the user see running processes, CPU and memory usage, and stop misbehaving programs.
- **Updates** — keeps the OS and applications up to date with security patches and bug fixes.

The user expects these to be available; bundling them with the OS saves the user from installing each separately.

## Program libraries

A **program library** is a collection of pre-written code (subroutines, classes, modules) that programs can use without having to write that code themselves.

Examples: a math library (`sin`, `cos`, `sqrt`); a network library (HTTP requests, socket I/O); a graphics library (drawing primitives, fonts); a database library (SQL access).

### Benefits

- **Saves time** — well-tested code is available off the shelf; the programmer focuses on the unique parts of their application.
- **Reliable** — library functions are normally well-tested and used by many other programs.
- **Standardised** — many programs use the same functions, with the same behaviour, so users get consistent results.

### Static vs dynamic libraries (background)

- A **static library** is linked into the executable at compile time. The library code is copied into the final binary. Pros: the executable runs standalone. Cons: larger executables; if the library is updated you must rebuild.
- A **dynamic (shared) library** (DLL on Windows, `.so` on Linux, `.dylib` on macOS) is loaded at run time. Pros: smaller executables, the library can be shared by many programs, updates apply to all programs at once. Cons: the system must have the right version available.

## Language translators

A programmer writes source code in a language; the computer can only execute machine code. The translator converts one into the other.

### Assembler

Translates **assembly language** into **machine code**. One assembly instruction maps to one (or a small number of) machine instructions. Needed whenever you write a low-level program in assembly (often used for tiny embedded systems, device drivers, optimised inner loops).

### Compiler

Translates a **high-level-language** program (Python, Java, C, C++, etc., though Python and Java vary — see below) into **machine code** (or sometimes into an intermediate form). The translation happens **once, before the program runs**.

- If errors are found during compilation, they are **all reported** and the program does not run. Once a clean executable is produced, the program can be **run many times** without re-translation.
- The output is a stand-alone **executable** (e.g. `.exe` on Windows). It can run **without the compiler being installed**.
- Generally **faster at run time** than interpreters (no translation overhead at runtime).
- Compiled code is tied to a specific CPU/OS — recompile for each platform.

### Interpreter

Translates a high-level-language program **one line at a time, and executes each line as it goes**. No standalone executable is produced.

- Errors are reported **as they are encountered** during execution — the program runs until it hits the first error, then **stops** (or reports and pauses).
- The programmer can **correct an error** and **continue** from that line — useful for development and learning.
- The **interpreter must be installed** on the machine that runs the program.
- Generally **slower at run time** — every execution re-translates each line.
- Easier to **port** — if the interpreter is available on multiple platforms, the same source code runs everywhere.

### Choosing between compiler and interpreter

**Use a compiler when**:

- You want **maximum runtime performance** (games, system software, scientific computing).
- You want to **distribute the program** to users who don't have the development tools installed.
- The program will be **run many times** so the one-time compile cost amortises.

**Use an interpreter when**:

- You want **fast development cycles** — write a line, run it, immediately see the result. Good for prototyping.
- The program needs to **run on many platforms** without recompiling.
- The program is **small** or **runs only once** (a quick script).
- Teaching beginners — the immediate feedback aids learning.

### Hybrid: Java (and similar)

Java illustrates a partial-compile + partial-interpret approach. Source code is **compiled into bytecode** (an intermediate, platform-independent language) by `javac`. At runtime, the **Java Virtual Machine (JVM)** **interprets** the bytecode (or **just-in-time-compiles** hot sections to native machine code). Benefits:

- Source compiles once into bytecode (catching many errors early).
- Bytecode runs on any platform with a JVM — "write once, run anywhere".
- Just-in-time compilation gives near-native performance for long-running programs.

Similar designs are used by C# (.NET), Python (CPython compiles to bytecode, then interprets), and many modern languages.

## Integrated Development Environment (IDE)

An **IDE** is a single application that brings together the tools needed to write, test and debug programs. Typical features:

- **Source code editor** with syntax highlighting (keywords, strings, comments shown in different colours), auto-indentation, line numbers, brace/bracket matching, code folding.
- **Auto-complete / context-sensitive prompts** — as you type, the IDE suggests variable names, function names, and method calls; it can show parameter information for function calls.
- **Code formatter / pretty-printer** — automatically lays out the code consistently.
- **Translator integration** — the compiler or interpreter is built in; you can compile/run with one keystroke; errors are shown inline in the code.
- **Debugger** — lets you:
  - set **breakpoints** to pause execution at chosen lines,
  - **step through** code line by line (step in, step over, step out),
  - **inspect variables** and the current call stack at any pause,
  - **watch** expressions and see how they change.
- **Version control integration** (git, SVN) — view file history, commit, branch, push without leaving the IDE.
- **Project management** — group files into projects, manage build settings.
- **Help system** — language reference, library documentation, examples.
- **Refactoring tools** — rename variables, extract methods, reorganise code safely.
- **Testing integration** — run unit tests, see pass/fail, jump to the failing assertion.

An IDE speeds development by combining all the tools and surfacing them through one interface, so the programmer can move from writing → running → debugging → fixing without switching applications. Common IDEs: Visual Studio, IntelliJ IDEA, PyCharm, Eclipse, Xcode, Visual Studio Code (with extensions). Less integrated alternatives include a plain text editor plus separate compiler and debugger — workable but slower.
