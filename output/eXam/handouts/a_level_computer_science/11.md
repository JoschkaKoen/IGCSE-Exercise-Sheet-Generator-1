## Programming basics

### From design to code

You should be able to translate a design — given as a **flowchart** or **structured English** — into pseudocode (and from there into a real programming language). Steps:

1. Identify variables and their types.
2. Translate input/output boxes into `INPUT` / `OUTPUT` statements.
3. Translate decision diamonds into `IF...ELSE...ENDIF` (or `CASE`).
4. Translate loop arrows into `WHILE`, `REPEAT...UNTIL`, or `FOR`.
5. Translate process boxes into assignment or calculation statements.
6. Check by tracing a small input.

### Constants and variables

A **constant** holds a value that **never changes** during the program. A **variable** holds a value that may change.

Declare constants and variables with their type:

```
CONSTANT Pi ← 3.14159
DECLARE Radius : REAL
DECLARE Area : REAL

Radius ← 5
Area ← Pi * Radius * Radius
```

Use constants for values with a clear meaning that recur — `Pi`, `MaxScore = 100`, `SchoolName = "Riverside High"`. They make code clearer and make later changes easy (update one declaration).

### Assignment and expressions

Use `←` for assignment:

```
Total ← Total + 1
Average ← Sum / Count
```

Expressions combine:

- **Arithmetic operators**: `+ - * / DIV MOD`. `DIV` is integer division; `MOD` is the remainder. `7 DIV 2 = 3`; `7 MOD 2 = 1`.
- **Comparisons**: `=`, `<>`, `<`, `>`, `<=`, `>=`.
- **Logical operators**: `AND`, `OR`, `NOT`.

Operator precedence (highest to lowest): `NOT` → `*`, `/`, `DIV`, `MOD` → `+`, `-` → comparisons → `AND` → `OR`. Use brackets when in doubt.

### Input and output

```
OUTPUT "Enter your name:"
INPUT Name
OUTPUT "Hello, ", Name
```

Multiple values can be printed on one line separated by commas (the implementation decides whether to add separators).

### Built-in functions and library routines

Many tasks have **library functions** ready-made, saving the trouble of writing them. Examples:

- String: `LENGTH(s)`, `LEFT(s, n)`, `RIGHT(s, n)`, `MID(s, start, len)`, `UCASE(s)`, `LCASE(s)`.
- Numeric: `INT(x)` (truncate), `ROUND(x)`, `ABS(x)`, `MOD(a, b)`, `RANDOM()` (random real in [0,1)).
- Type conversion: `STR(x)` (number to string), `VAL(s)` (string to number — fails if `s` isn't numeric).

Always check the question paper's reference for the exact names — Cambridge uses a fixed list.

## Selection constructs

### Simple IF

```
IF condition THEN
    statements
ENDIF
```

### IF...ELSE

```
IF age >= 18 THEN
    OUTPUT "Adult"
ELSE
    OUTPUT "Minor"
ENDIF
```

### Nested IF

When you need more than two cases, you can nest:

```
IF mark >= 80 THEN
    grade ← "A"
ELSE
    IF mark >= 60 THEN
        grade ← "B"
    ELSE
        IF mark >= 40 THEN
            grade ← "C"
        ELSE
            grade ← "F"
        ENDIF
    ENDIF
ENDIF
```

Nesting deeper than two or three levels becomes hard to read. Consider using a `CASE` statement.

### CASE

```
CASE OF Grade
    "A": OUTPUT "Excellent"
    "B": OUTPUT "Good"
    "C": OUTPUT "Pass"
    OTHERWISE: OUTPUT "Try again"
ENDCASE
```

`CASE` is cleaner than nested IFs when you are testing a single value against several specific options. Cambridge's `CASE` typically supports single values (`"A": ...`), value lists (`1, 2, 3: ...`), and ranges (`1 TO 5: ...`).

## Iteration constructs

Three loop types, with different conditions about how many times the body runs.

### Count-controlled (FOR) loop

Use when you know exactly **how many times** to repeat (or a clear start/stop/step).

```
FOR i ← 1 TO 10
    OUTPUT i
NEXT i
```

With a step:

```
FOR i ← 10 TO 1 STEP -1
    OUTPUT i
NEXT i
```

Best for: iterating a fixed number of times; processing each element of an array; counting.

### Pre-condition (WHILE) loop

Tests the condition **before** each iteration. May execute **zero** times.

```
WHILE total < 100 DO
    INPUT n
    total ← total + n
ENDWHILE
```

Best for: "keep doing this while X is true"; processing input until a sentinel; running until a result is good enough.

### Post-condition (REPEAT...UNTIL) loop

Tests the condition **after** each iteration. Always executes **at least once**.

```
REPEAT
    INPUT password
UNTIL password = correctPassword
```

Best for: "do this and check whether you're done"; menu-driven programs that always show the menu at least once; reading the first input then deciding whether to keep going.

### Choosing the right loop

- Iterations known up front → **FOR**.
- May need zero iterations → **WHILE**.
- Always at least one iteration → **REPEAT...UNTIL**.

Justify your choice by referring to whether the iteration count is known, and whether the body must run at least once. A typical exam question gives a scenario ("validate password until correct, but always prompt at least once") and asks which loop fits best. Match the scenario's properties to the loop's semantics.

## Structured programming: procedures and functions

A **structured program** is built from small, named **subroutines** that each have a single responsibility. There are two kinds of subroutine.

### Procedure

A **procedure** is a named block of code that performs an action. It is **called** by name; it may take **parameters** but it does **not return a value** (it returns control to the caller after running).

```
PROCEDURE Greet(name : STRING)
    OUTPUT "Hello, ", name
ENDPROCEDURE

// later:
CALL Greet("Ada")
```

### Function

A **function** is like a procedure but it **returns a value**. The return value becomes part of an expression.

```
FUNCTION Square(x : INTEGER) RETURNS INTEGER
    RETURN x * x
ENDFUNCTION

// later:
result ← Square(5) + 1     // result = 26
```

### When to use a procedure vs a function

- Use a **procedure** when the subroutine **performs an action** (writes to a file, updates a record, displays a menu) but doesn't compute a value to be used by the caller.
- Use a **function** when the subroutine **computes a value** that the caller will use (a maximum, a sum, a transformation of an input).

### Parameters

**Parameters** are the variables a subroutine declares to receive input from the caller. When the caller passes values in those positions, the values are called **arguments**.

Two ways parameters can pass:

- **By value** — the called routine receives a **copy** of the argument; changes to it inside the routine do not affect the caller's variable. Use for inputs that the routine reads but doesn't update.
- **By reference** — the routine receives a **reference** to the caller's variable; changes inside the routine **do** affect the caller. Use when the routine must update one of its parameters.

In Cambridge pseudocode:

```
PROCEDURE Swap(BYREF a : INTEGER, BYREF b : INTEGER)
    DECLARE temp : INTEGER
    temp ← a
    a ← b
    b ← temp
ENDPROCEDURE
```

### Local vs global variables

- A **local variable** is declared inside a subroutine and exists only while that subroutine is running.
- A **global variable** is declared outside any subroutine and is visible everywhere.

Prefer **local variables** and parameters. Heavy use of global variables makes code hard to understand (any subroutine could change them) and hard to test. Use globals sparingly, for genuinely program-wide settings.

### Where to use subroutines

Use a subroutine whenever:

- the same logic appears in **more than one place** — extract it once and call it.
- a block of code has a **distinct purpose** that can be named — improves readability even if used only once.
- the program is **complex** — decomposing it into named subroutines makes it manageable.
- you are **testing a piece in isolation** — small subroutines are easier to test than monolithic code.

Avoid making subroutines so small that calling overhead exceeds their content; small, single-purpose subroutines that are reused or improve readability are the sweet spot.

### Subroutine terminology

- **Definition** — the code that defines the subroutine (the `PROCEDURE ... ENDPROCEDURE` block).
- **Call** — the place where the subroutine is invoked.
- **Argument** — value supplied by the caller in a call.
- **Parameter** — variable declared by the subroutine to receive arguments.
- **Return value** (functions only) — value passed back to the caller.
- **Signature** — name + parameter list + return type. Defines how callers see the routine.
- **Scope** — the region of code in which a name is visible.

## Writing efficient pseudocode

Some habits that make pseudocode (and the resulting program) cleaner and faster:

- **Move invariants out of loops** — if a calculation inside a loop doesn't change with the loop variable, do it once before the loop.

  ```
  // bad
  FOR i ← 1 TO 100
      total ← total + (price * tax_rate)   // (price * tax_rate) is constant
  NEXT i

  // better
  unit ← price * tax_rate
  FOR i ← 1 TO 100
      total ← total + unit
  NEXT i
  ```

- **Exit a loop early** when the answer is found — use `EXIT FOR` / `EXIT WHILE` to stop a linear search the moment the target is located.
- **Avoid redundant work** — don't recompute values; store the result in a variable and reuse it.
- **Choose the right data structure** — using an array is much faster than a list of separate variables when the items belong together.
- **Replace deeply nested IF chains with CASE** when testing one variable against many values.
- **Decompose** — break long subroutines into smaller ones with clear names.
- **Comment intent, not mechanics** — `// validate the postcode` is more useful than `// loop 6 times`.
- **Use meaningful identifier names** — `numberOfPupils`, not `n`.
- **Initialise variables** before use — never assume the runtime gives them a value.
- **Validate inputs** at the boundary; don't trust raw input deeper in the code.
