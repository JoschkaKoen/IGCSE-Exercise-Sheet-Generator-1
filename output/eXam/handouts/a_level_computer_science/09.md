## Computational thinking

Computational thinking is the set of mental tools used to **analyse a problem and design a solution that a computer can execute**. Two of the most useful are abstraction and decomposition.

### Abstraction

**Abstraction** is the process of **focusing on the essential features** of a problem (or system) and **ignoring the irrelevant detail**. The result is a simpler model that captures what matters.

Examples:

- A **map of a train network** abstracts away geography, distances and street names, keeping only the stations and the lines connecting them. That is enough to plan a journey.
- A **class in object-oriented programming** abstracts a real-world entity (a customer, a vehicle) into a small set of attributes and methods that the rest of the system cares about. Other details (how the customer dresses, what the vehicle weighs) are left out.
- A **function** abstracts a piece of work behind a name and a parameter list — callers care about what it does, not how.

Abstraction is essential because a complete model of any real-world problem would be too large to reason about. By picking the right abstraction you make the problem tractable.

### Decomposition

**Decomposition** is **breaking a large problem into smaller sub-problems**, each of which is easier to solve. The smaller pieces are independent (or nearly so), and can be tackled one at a time, perhaps by different people.

Steps:

1. Identify the main components of the task.
2. Break each component into smaller sub-tasks.
3. Continue until each sub-task is small enough to design directly.
4. Solve the small tasks; combine their solutions into the full solution.

For a shop's stock-control program:

- top level: "manage stock"
- one level down: "record sales", "record deliveries", "produce reports", "reorder low-stock items"
- next level (for "record sales"): "look up product", "decrease stock count", "save the transaction"

Decomposition makes large problems manageable, lets work be divided across a team, and produces modular code that is easier to test and maintain.

## Algorithms — the idea

An **algorithm** is a **solution to a problem expressed as a sequence of defined steps**. Each step is **unambiguous** (one interpretation), **deterministic** (the same input always produces the same output), **finite** (the steps end), and **effective** (each can be carried out).

An algorithm describes **what to do**, but is independent of the **programming language** in which it is finally implemented. The same algorithm can be coded in Python, Java, C, pseudocode, or carried out on paper.

## Identifier table

When you start an algorithm, list every piece of data the problem uses with three pieces of information about each — this is the **identifier table**:

| Variable name | Data type | Description |
|---|---|---|
| `Category` | `STRING` | the product category (e.g. `"Fruit"`) |
| `SaleDate` | `DATE` | when the item was sold |
| `ItemCost` | `REAL` | cost of the item (e.g. `12.67`) |
| `InStock` | `BOOLEAN` | `TRUE` if the item is in stock |

Conventions:

- Variable names should be **descriptive** (`ItemCost`, not `x`), use **camelCase** or **PascalCase**, and avoid spaces.
- Data types in pseudocode are usually `INTEGER`, `REAL`, `STRING`, `CHAR`, `BOOLEAN`, `DATE`, plus structured types like arrays.

The identifier table forms the link between the problem description and the algorithm — it forces you to name and classify every relevant piece of data before writing any code.

## Pseudocode — the three basic constructs

**Pseudocode** is a structured, programming-language-neutral notation for describing algorithms. Cambridge uses a defined set of pseudocode conventions.

### 1. Sequence

Steps executed one after another, top to bottom.

```
INPUT Name
INPUT Age
OUTPUT "Hello", Name
```

### 2. Selection (decisions)

A choice of which steps to execute, based on a condition.

```
IF Age >= 18 THEN
    OUTPUT "Adult"
ELSE
    OUTPUT "Minor"
ENDIF
```

For more options:

```
CASE OF Grade
    "A": OUTPUT "Excellent"
    "B": OUTPUT "Good"
    "C": OUTPUT "Pass"
    OTHERWISE: OUTPUT "Fail"
ENDCASE
```

### 3. Iteration (repetition / loops)

Repeat a block of steps. Three main forms:

- **FOR** loop — a counted loop with a known number of iterations:

```
FOR i ← 1 TO 10
    OUTPUT i
NEXT i
```

- **WHILE** loop — repeat while a condition is true; the condition is tested **before** each iteration. May execute zero times.

```
total ← 0
WHILE total < 100 DO
    INPUT n
    total ← total + n
ENDWHILE
```

- **REPEAT...UNTIL** loop — the condition is tested **after** each iteration; the body always runs at least once.

```
REPEAT
    INPUT n
UNTIL n > 0
```

### Common pseudocode operations

- **Assignment**: `x ← 5` (an arrow; `=` is reserved for comparison).
- **Input**: `INPUT variable`.
- **Output**: `OUTPUT expression`.
- **Comparison**: `=`, `<>`, `<`, `>`, `<=`, `>=`.
- **Logical operators**: `AND`, `OR`, `NOT`.
- **Arithmetic**: `+ - * / DIV MOD` (where `DIV` is integer division and `MOD` is remainder).
- **Strings**: `LENGTH(s)`, `LEFT(s, n)`, `RIGHT(s, n)`, `MID(s, start, length)`, `s1 & s2` for concatenation.

### Input → Process → Output

Every program follows this structure at some level. Pseudocode makes it explicit:

```
// INPUT
INPUT Length
INPUT Width

// PROCESS
Area ← Length * Width
Perimeter ← 2 * (Length + Width)

// OUTPUT
OUTPUT "Area = ", Area
OUTPUT "Perimeter = ", Perimeter
```

Beginners often jump straight to processing — taking a moment to list the inputs and outputs first makes the algorithm cleaner.

## Documenting an algorithm: three notations

The same algorithm can be expressed in three forms — choose what suits the situation.

### Structured English

A natural-language description with **indentation** to show structure and **fixed keywords** for control flow. Useful for high-level descriptions before pseudocode.

```
Ask the user for two numbers.
If the first number is bigger than the second:
    Output the first number.
Otherwise:
    Output the second number.
```

### Flowchart

A diagram with standard shapes connected by arrows.

| Shape | Meaning |
|---|---|
| Rounded rectangle | Start / Stop |
| Parallelogram | Input / Output |
| Rectangle | Process (e.g. assignment, calculation) |
| Diamond | Decision (yes/no branching) |
| Arrow | Flow of control |

Iterations are drawn by looping arrows back to an earlier point. Decisions branch into two arrows for "Yes" and "No".

### Pseudocode

The structured-keyword notation already described. Best for precise expression that is close to code.

You should be able to **convert between any pair** of these three. Steps:

- structured-English → pseudocode: replace English keywords with formal ones, add formal `IF / ENDIF`, `WHILE / ENDWHILE` markers.
- structured-English → flowchart: map each sentence to a flowchart symbol; show branches as decision diamonds.
- pseudocode ↔ flowchart: each `IF` is a decision diamond; each loop is a back-arrow; sequences are stacked rectangles.

## Stepwise refinement

**Stepwise refinement** is the process of starting with a **high-level outline** and **progressively expanding** each step into more detail, until every step is small enough to translate directly into code.

Example: working out an average of $n$ numbers.

**Level 1** (high level):

```
Read in the numbers
Compute the average
Output the average
```

**Level 2** (refine):

```
INPUT n
total ← 0
FOR i ← 1 TO n
    Read next number; add to total
NEXT i
average ← total / n
OUTPUT average
```

**Level 3** (refine each subtask if needed):

```
INPUT n
total ← 0
FOR i ← 1 TO n
    INPUT value
    total ← total + value
NEXT i
average ← total / n
OUTPUT average
```

Each refinement preserves the structure of the previous level and adds detail where needed. Stepwise refinement keeps the design focused: you can't get the small details right if the overall flow is wrong.

## Logic statements

A **logic statement** is a Boolean condition used to control branching. It is built from:

- comparisons: `x > 10`, `name = "Ada"`,
- logical connectives: `AND`, `OR`, `NOT`,
- grouping: parentheses.

Use logic statements as the condition of `IF`, `WHILE`, `REPEAT...UNTIL`. For example:

```
IF age >= 18 AND hasLicence = TRUE THEN
    OUTPUT "Can drive"
ENDIF
```

```
WHILE attempts < 3 AND NOT loggedIn DO
    INPUT password
    IF password = correctPassword THEN
        loggedIn ← TRUE
    ELSE
        attempts ← attempts + 1
    ENDIF
ENDWHILE
```

Precedence (from highest to lowest): `NOT`, then `AND`, then `OR`. Use brackets when in doubt to keep the meaning clear:

```
(age >= 18 OR hasParentConsent) AND hasTicket
```

Beware of common mistakes:

- `a = 1 OR 2` is wrong — write `a = 1 OR a = 2`.
- `NOT a > 5` means `NOT (a > 5)` — i.e. `a <= 5`. Don't write `a NOT > 5`.
- Combining negations: `NOT (A AND B)` is the same as `(NOT A) OR (NOT B)` (De Morgan's law) — useful for simplifying nested conditions.
