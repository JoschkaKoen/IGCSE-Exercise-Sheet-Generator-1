## Searching algorithms

A **search** algorithm looks for a given **target value** in a collection (often an array) and returns its position (or "not found").

### Linear search

Walk through the collection from start to end, comparing each element with the target.

```
FOR i ← 1 TO n
    IF A[i] = target THEN
        RETURN i
    ENDIF
NEXT i
RETURN -1   // not found
```

- **No preparation** of the data needed; works on any list.
- **Worst case**: target is at the end or not present — must look at all $n$ elements: O($n$).
- **Average case**: about $n/2$ comparisons.
- **Best case**: target at the first position — 1 comparison.

Use linear search on **unsorted** data, or when the collection is too small to bother sorting first.

### Binary search

Requires the data to be **sorted** in advance.

The idea: look at the middle element. If it is the target, done. If the target is smaller, search the left half; if larger, search the right half. Halve the search range with each comparison.

```
low ← 1
high ← n
WHILE low <= high DO
    mid ← (low + high) DIV 2
    IF A[mid] = target THEN RETURN mid
    IF A[mid] < target THEN
        low ← mid + 1
    ELSE
        high ← mid - 1
    ENDIF
ENDWHILE
RETURN -1   // not found
```

- Requires **sorted** data.
- **Worst case**: O($\log_{2} n$) comparisons. For $n = 1{,}000{,}000$, only about 20 comparisons.
- **Much faster** than linear search on large sorted arrays.
- **Cost**: sorting the data first (one-off) takes O($n \log n$). Worth it if you'll search many times.

## Sorting algorithms

**Sorting** rearranges a collection into a defined order (ascending or descending by some key).

### Bubble sort

Repeatedly walk through the array; on each pass, **swap adjacent pairs** that are in the wrong order. The largest element "bubbles" to the end after the first pass, the second-largest after the second pass, and so on.

```
FOR pass ← 1 TO n - 1
    swapped ← FALSE
    FOR i ← 1 TO n - pass
        IF A[i] > A[i + 1] THEN
            // swap A[i] and A[i+1]
            temp ← A[i]
            A[i] ← A[i + 1]
            A[i + 1] ← temp
            swapped ← TRUE
        ENDIF
    NEXT i
    IF swapped = FALSE THEN EXIT FOR   // already sorted, stop early
NEXT pass
```

- **Best case** (already sorted): O($n$) with the early-exit optimisation.
- **Worst case** / **average**: O($n^{2}$).
- Simple to understand and code; not efficient for large $n$.

### Insertion sort

Build up a sorted prefix from the left. For each new element, **insert** it into the correct position within the already-sorted prefix by shifting larger elements to the right.

```
FOR i ← 2 TO n
    key ← A[i]
    j ← i - 1
    WHILE j >= 1 AND A[j] > key DO
        A[j + 1] ← A[j]
        j ← j - 1
    ENDWHILE
    A[j + 1] ← key
NEXT i
```

- **Best case** (already sorted): O($n$) — the inner loop never iterates.
- **Worst case**: O($n^{2}$).
- Good for **small** arrays or **nearly-sorted** arrays.
- Sorts the array **in place** (no extra arrays needed), and is **stable** (preserves the order of equal elements).

### Tracing a sort

A common exam task: given an array and a sort algorithm, produce a table showing the array's contents **after each pass** of the outer loop. Steps:

1. Write down the initial array.
2. Trace one outer-loop iteration at a time — note which swaps or insertions happen.
3. Write the array state at the end of each outer iteration.

A row per outer iteration is usually expected; some questions also want the values of the loop variables (`i`, `j`, `key`) at key moments.

For example, sorting `['D', 'T', 'H', 'R']` with insertion sort:

- Initial: `[D, T, H, R]`.
- Pass 1 (i=2, key='T'): T is already in order — no shifts. `[D, T, H, R]`.
- Pass 2 (i=3, key='H'): T > H, shift; D < H, stop. `[D, H, T, R]`.
- Pass 3 (i=4, key='R'): T > R, shift; H < R, stop. `[D, H, R, T]`.

## Abstract Data Types in algorithms

The ADTs you met in Topic 10 — **stack, queue, linked list** — appear inside many algorithms.

- A **stack** drives depth-first traversal, undo history, function-call return addresses.
- A **queue** drives breadth-first traversal, print-job ordering.
- A **linked list** lets data grow and shrink without copying, and underpins more elaborate structures.

### ADTs built from other ADTs

ADTs can be **implemented in terms of other ADTs** — not just from raw arrays.

- A **queue** can be implemented with **two stacks**: one for incoming items, one for outgoing. To enqueue, push onto the incoming stack. To dequeue, if the outgoing stack is empty, pop everything from the incoming stack to the outgoing (reversing order); then pop the top of the outgoing stack.
- A **stack** can be implemented with a **linked list**: each push prepends a new head node; each pop removes the head.
- A **queue** can be implemented with a **linked list** with two pointers (head and tail) — enqueue at the tail, dequeue from the head.
- A **binary tree** can be implemented with linked lists of nodes that each carry two child pointers.

Layering ADTs this way **separates concerns**: the algorithm using the ADT does not care how it is implemented, as long as the interface (push/pop, enqueue/dequeue) works.

## Comparing algorithms

Two algorithms can both solve the same problem but at different cost. Use objective criteria to compare them.

### Time complexity

How long does the algorithm take, as a function of the input size $n$? Often expressed in **Big-O** notation (the dominant term as $n$ grows). Examples:

- O(1) — constant time.
- O($\log n$) — logarithmic — binary search.
- O($n$) — linear — linear search, summing an array.
- O($n \log n$) — efficient sorts (merge sort, quicksort).
- O($n^{2}$) — bubble sort, insertion sort.

Smaller order = better at scale, even if for small $n$ another algorithm runs faster.

Also useful in practice: **best case**, **worst case** and **average case** for a given algorithm.

### Memory complexity / space complexity

How much extra memory does the algorithm need? Bubble sort and insertion sort use O(1) extra memory ("in place"); merge sort uses O($n$) extra memory for the temporary array; recursion uses memory on the call stack proportional to recursion depth.

A trade-off often exists between time and memory — a faster algorithm may need more memory and vice versa.

### Other criteria

- **Simplicity** — easier to code, easier to maintain, less risk of bugs.
- **Stability** — does it preserve the relative order of equal-key elements? Important when sorting by one key while preserving an earlier sort by another.
- **Adaptiveness** — does it run faster on already-sorted or nearly-sorted data?

In a real situation, **the right algorithm depends on the data and the constraints** — array size, distribution, available memory, real-time requirements.

## Recursion

A **recursive** algorithm calls itself with a smaller version of the same problem. Each call works on a smaller input than the one above it; at some point a **base case** ends the chain without further recursion.

### A recursive function has two parts

1. **Base case** — input is small enough to solve directly without further recursion. Without a base case, the recursion never stops.
2. **Recursive case** — input is reduced (e.g. smaller number, shorter list, smaller sub-tree), and the function calls itself with the reduced input.

### Examples

**Factorial**:

```
FUNCTION Factorial(n : INTEGER) RETURNS INTEGER
    IF n = 0 OR n = 1 THEN
        RETURN 1              // base case
    ELSE
        RETURN n * Factorial(n - 1)   // recursive case
    ENDIF
ENDFUNCTION
```

**Fibonacci** (slow recursive form):

```
FUNCTION Fib(n : INTEGER) RETURNS INTEGER
    IF n <= 1 THEN
        RETURN n
    ELSE
        RETURN Fib(n - 1) + Fib(n - 2)
    ENDIF
ENDFUNCTION
```

**Binary search**:

```
FUNCTION BinSearch(A, low, high, target) RETURNS INTEGER
    IF low > high THEN
        RETURN -1
    ELSE
        mid ← (low + high) DIV 2
        IF A[mid] = target THEN RETURN mid
        IF A[mid] < target THEN
            RETURN BinSearch(A, mid + 1, high, target)
        ELSE
            RETURN BinSearch(A, low, mid - 1, target)
        ENDIF
    ENDIF
ENDFUNCTION
```

### When recursion is natural

- Problems with a **self-similar** structure: trees, fractals, divide-and-conquer (binary search, merge sort, quicksort), traversing nested data.
- Problems definable in terms of smaller instances of themselves.

When recursion is a poor fit, an explicit loop is usually cleaner and faster.

### Tracing a recursive call

A common task: trace `Factorial(4)`.

| Call | Returns |
|---|---|
| `Factorial(4)` | `4 * Factorial(3)` |
| `Factorial(3)` | `3 * Factorial(2)` |
| `Factorial(2)` | `2 * Factorial(1)` |
| `Factorial(1)` | `1` |
| ... unwinds ... | |
| `Factorial(2)` | `2 * 1 = 2` |
| `Factorial(3)` | `3 * 2 = 6` |
| `Factorial(4)` | `4 * 6 = 24` |

Final result: `24`. Track each pending call on a stack to keep your trace clear.

### Risks of recursion

- **Infinite recursion** if the base case is missed or unreachable — eventually crashes with a stack overflow.
- **High memory use** for deep recursion (one stack frame per pending call).
- **Slow** if the recursion does redundant work (naive Fibonacci is exponential time because it re-computes the same subproblems many times — use iteration or memoisation).

## What a compiler does for recursive code

Recursion relies on **each call having its own copy** of the local variables and parameters, separate from earlier calls. The compiler arranges this on the **call stack**.

For each call:

1. **Push a new stack frame** onto the call stack. The frame contains:
   - the values of the function's **parameters**,
   - the function's **local variables**,
   - the **return address** (where to resume in the caller),
   - sometimes the previous **frame pointer**.
2. Begin executing the function with the new frame's local context.

When the function returns:

1. The return value is placed where the caller can find it (in a register or on the stack).
2. The frame is **popped** off the stack — its local variables disappear.
3. Control resumes at the **return address** saved in the frame.

Because each call has its own frame, recursive calls don't trample each other's variables — exactly what recursion needs. The call stack can grow large for deep recursion, which is why very deep recursion may exceed the stack's allocated memory and crash.

This stack-based call mechanism is exactly the same one the compiler uses for non-recursive function calls — recursion just exposes the call-stack machinery more visibly. There is no special "recursion mechanism" — the compiler generates ordinary call-and-return code; the program does the rest by calling itself.
