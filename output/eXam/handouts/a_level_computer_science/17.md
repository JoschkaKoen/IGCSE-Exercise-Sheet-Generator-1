## How encryption works

**Encryption** transforms readable data (**plaintext**) into an unreadable form (**ciphertext**) using a mathematical operation that depends on a **key**. Only someone with the correct key can reverse the operation — **decryption** — to recover the original plaintext.

An attacker who intercepts the ciphertext without the key sees only meaningless data. The mathematics behind modern encryption is designed so that "trying every possible key" is computationally infeasible in any practical amount of time.

### Symmetric encryption

The **same key** is used for both encryption and decryption. Sender and receiver must both hold the secret key.

- **Fast** — symmetric algorithms (AES, ChaCha20) are efficient even on huge volumes of data.
- Suited to **bulk data** — encrypting a whole disk, a video stream, a long message.
- **Problem**: how do you share the key securely in the first place? If you send the key over the internet, an attacker who intercepts it can read everything. Solving this **key distribution problem** is where asymmetric encryption comes in.

### Asymmetric encryption (public-key cryptography)

Each user has a **pair of keys** that are mathematically related:

- a **public key** that they can publish openly,
- a **private key** that they keep secret.

The key pair is generated so that:

- data encrypted with the **public key** can be decrypted **only with the matching private key**, and
- data encrypted with the **private key** can be decrypted **only with the matching public key**.

To send a secret message to Alice:

1. Get Alice's **public key** (it's published, anyone can fetch it).
2. Encrypt the message with Alice's public key.
3. Send the ciphertext.
4. Only Alice — the holder of the matching **private key** — can decrypt.

No prior key exchange is needed. Asymmetric algorithms (RSA, elliptic-curve) are mathematically based on hard problems (factoring large integers, discrete logarithms in elliptic-curve groups).

The trade-off: asymmetric encryption is **much slower** than symmetric — perhaps a few thousand times. Not suited to encrypting large amounts of data.

### Hybrid approach (used by almost every real system)

Use asymmetric encryption to securely exchange a fresh symmetric key, then use the symmetric key to encrypt the actual data.

1. Client generates a random symmetric **session key**.
2. Client encrypts the session key with the server's **public key**.
3. Server decrypts it with its **private key**.
4. Both ends now share the symmetric session key.
5. The bulk of the conversation uses **symmetric encryption** with that session key — fast.
6. The session key is discarded when the session ends.

This is how HTTPS, SSH and many other protocols work in practice.

### Hashing (closely related but not encryption)

A **cryptographic hash function** takes any input and produces a fixed-size output (the **hash digest** or **fingerprint**) such that:

- the same input always produces the same digest,
- it is computationally infeasible to construct two different inputs that produce the same digest (**collision-resistance**),
- it is infeasible to find an input that hashes to a given digest (**preimage-resistance**),
- a tiny change in input produces a wildly different digest (**avalanche effect**).

Hashing is **one-way** — you cannot get the original input back from the digest. Hashes are not encryption, but they are used in similar contexts: storing password verifiers, ensuring data integrity, building digital signatures.

## SSL / TLS

**SSL (Secure Sockets Layer)** and its successor **TLS (Transport Layer Security)** are **protocols** that provide encryption and authentication for data sent over a network. TLS has long replaced SSL in practice; the names are often used interchangeably.

### Functions of SSL/TLS

- **Encrypt** the data in transit so eavesdroppers see only ciphertext.
- **Authenticate** the server (and optionally the client) using digital certificates — you connect to the server you think you connect to.
- Provide **data integrity** — detect any tampering with the data in transit.

### How TLS works (outline)

1. The client connects to the server and proposes a list of cipher suites it supports.
2. The server picks one cipher suite and sends its **digital certificate** (containing its public key, signed by a Certificate Authority).
3. The client verifies the certificate (Authority signature valid? not expired? hostname matches?).
4. The two ends perform a **key-exchange** (using asymmetric crypto) to derive a fresh shared **session key**.
5. All subsequent traffic uses **symmetric encryption** with the session key.

The result: an encrypted, authenticated, integrity-checked tunnel that higher-level protocols (HTTP, SMTP, IMAP) can run on top of.

### Where SSL/TLS is appropriate

- **HTTPS** — secure web browsing (banking, shopping, email login, social media). Default for almost every site today.
- **Secure email** between mail servers and between mail client and server (IMAPS, SMTPS, POP3S).
- **Online banking and payments** — sensitive financial data.
- **VPN** — some VPNs (OpenVPN, IKEv2) build on TLS.
- **Voice / video calls** that require confidential signalling.
- **Any service that handles personal data**, login credentials, or payment information.

A common exam question: "Give two examples of where SSL/TLS would be appropriate." Pick scenarios where **sensitive information is transmitted** — banking, online shopping, login pages.

## Digital certificates

A **digital certificate** is an electronic document that **binds an identity (a domain name, an organisation) to a public key**. It is issued and signed by a trusted **Certificate Authority (CA)**.

### Contents of a typical certificate

- the **subject** — who the certificate identifies (e.g. the domain `www.example.com`),
- the subject's **public key**,
- the **issuer** — the CA that signed it (e.g. Let's Encrypt, DigiCert),
- a **serial number**,
- **validity period** — not-before and not-after dates,
- the **digital signature** of the CA over all the above.

### How a certificate is verified

The browser (or other client) holds a list of **trusted root CAs** (their certificates ship with the browser/OS).

1. The server presents its certificate.
2. The client checks the **expiry dates**.
3. The client checks the **subject name** matches the URL.
4. The client checks the certificate is **signed by a trusted CA** — using the CA's public key (already in the trust store) to verify the signature.
5. If the certificate is signed by an intermediate CA, the client follows the **certificate chain** up to a root CA in the trust store.
6. If anything fails, the browser shows a warning (the familiar "Your connection is not private" page).

When a certificate verifies cleanly, the client knows:

- the server's claimed identity has been vetted by a trusted CA,
- the server's public key inside the certificate genuinely belongs to that identity,
- the certificate is current.

### Digital signatures

A **digital signature** is a cryptographic operation that proves the signer's identity **and** that the signed data has not been changed.

To sign a message:

1. Compute a **cryptographic hash** of the message.
2. **Encrypt the hash with the sender's private key** — that encrypted hash is the signature.
3. Send the message and the signature (and the sender's public-key certificate).

To verify:

1. Compute the hash of the received message using the same algorithm.
2. **Decrypt the signature with the sender's public key** — yielding the hash the sender computed.
3. Compare the two hashes. If they match:
   - The signature was made by someone holding the sender's private key (**authentication**).
   - The message has not been changed in transit (**integrity**), because any change would alter the hash and the comparison would fail.

A signature does **not encrypt** the message — anyone can read the message. It just proves authorship and integrity. If you need both confidentiality and authenticity, encrypt the message **and** sign it.

### Putting it together

A typical secure web request:

1. Client connects to `https://www.bank.com`.
2. Server sends its digital certificate.
3. Client verifies the certificate against trusted CAs.
4. Client uses the server's public key (from the certificate) to securely exchange a session key.
5. Client and server exchange data, encrypted with the session key, with integrity ensured by TLS.

Throughout the session, encryption protects against eavesdroppers, certificates prove the server's identity, and integrity checks ensure no MITM has altered the data in flight.
