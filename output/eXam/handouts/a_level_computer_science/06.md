## Security, privacy and integrity — three distinct ideas

These three words sound alike but mean different things.

- **Security** — protection of data from **unauthorised access, modification or destruction**. Concerned with who can see and change the data, and with keeping it safe from threats.
- **Privacy** — the right of an individual or organisation to **control who sees their personal data**. Concerned with **consent** and **purpose**: only authorised people should view data, and only for the agreed reasons.
- **Integrity** — the **accuracy and completeness** of data. Concerned with the data being **correct** — not corrupted, not accidentally changed, not invalid.

A file can be secure (only the right people can access it) but lack integrity (a typo corrupted it). It can have high integrity (the values are all correct) but lack privacy (anyone can read it). All three are needed.

## Why security matters

Two things to protect:

- **Security of data** — keeping the data itself confidential, intact, and available. Loss of data could mean financial loss, identity theft, reputational damage, legal action.
- **Security of the computer system** — keeping the hardware and software functional. A compromised system can be used to attack others, harvest credentials, or hold the user to ransom.

Both must be defended in any modern computing environment.

## Threats from networks and the internet

Common threats:

- **Malware** — broad term for malicious software:
  - **Virus** — self-replicating code that attaches itself to other programs and spreads when those programs run.
  - **Worm** — self-replicating code that spreads automatically over networks without needing user action.
  - **Trojan horse** — a program that masquerades as something useful but contains malicious code that activates when the user runs it.
  - **Spyware** — secretly collects information about the user (keystrokes, browsing, passwords) and sends it to an attacker.
  - **Ransomware** — encrypts the user's files and demands payment to decrypt them.
  - **Adware** — pushes unwanted advertisements; may also track the user.
- **Phishing** — fake emails/websites that trick users into entering credentials or downloading malware. **Spear phishing** is targeted; **pharming** redirects users to fake sites by tampering with DNS.
- **Hacking** — unauthorised access to a computer or network, typically by exploiting software vulnerabilities or weak passwords.
- **Denial of Service (DoS) / Distributed DoS (DDoS)** — overwhelms a server with traffic so legitimate users cannot reach it.
- **Eavesdropping (interception)** — capturing data as it is transmitted; especially a risk on unencrypted Wi-Fi and the open internet.
- **Man-in-the-middle (MITM)** — attacker secretly relays / alters communication between two parties who believe they are talking directly.
- **Social engineering** — manipulating people into giving up information (pretending to be IT support, finding personal details to guess passwords).

## Security measures: from standalone PC to network

Defences scale with the environment.

### A standalone PC

- **Strong password / passphrase** to log in. Don't reuse across services.
- **Antivirus / antimalware** software running and kept up to date.
- **Software updates** applied promptly — vulnerabilities are patched.
- **Backup** important data to separate media (or cloud) so ransomware or hardware failure doesn't wipe it.
- **Encryption of the disk** (full-disk encryption) so a stolen laptop is not a privacy disaster.
- **Locked screen** when stepping away.

### A networked PC

All of the above, plus:

- **Firewall** at the network boundary and on each PC, filtering inbound and outbound traffic.
- **User-level permissions** — each user has their own login; admin rights are reserved for genuine admins; not every user is root.
- **Centralised account management** (e.g. Active Directory) so accounts can be created, disabled or password-reset for everyone in one place.
- **Audit logs** — record who logged in, what files they touched.

### Inter-network and the internet

- **VPN (Virtual Private Network)** — encrypts traffic between the user and the corporate gateway so the link across the public internet appears to be a private wire.
- **HTTPS** — encrypts web traffic between browser and server.
- **TLS / SSL** — the underlying cryptographic protocols used by HTTPS, secure email, etc.
- **Intrusion-detection / intrusion-prevention systems** — analyse network traffic for known attack patterns.

## Methods to restrict risks

For each threat you should know one or more countermeasures:

- **Unauthorised interception of data in transit** → **encrypt** the data (HTTPS for web, VPN for remote access, SSH for shell access). Even if intercepted, the encrypted data is meaningless without the key.
- **Unauthorised access to a system** → **strong authentication** (long passwords; **two-factor authentication** using a code from your phone or a security key); user permissions; lock-outs after several failed login attempts.
- **Malware** → **antivirus / antimalware** scanning and real-time monitoring; keep the OS and applications **patched**; avoid downloading software from untrusted sources; restricted file-type policies on email gateways.
- **Phishing** → user training to spot suspicious emails; **email filtering**; not clicking unverified links; verifying the URL before entering credentials. Email server can check **SPF/DKIM/DMARC** records to spot forged senders.
- **Internal threats / accidents** → **least-privilege** principle (give each user only what they need); **separation of duties**; auditing.
- **DDoS** → **rate limiting**, traffic filtering, **content-distribution networks** that can absorb bursts.

## Methods designed to protect the security of data

Several methods that go beyond the user-facing defences:

- **Encryption** — convert plaintext into ciphertext using a key. Symmetric encryption (e.g. AES) uses one shared secret key; asymmetric (e.g. RSA) uses a public/private key pair. Encryption protects data both **at rest** (on disk) and **in transit** (on the network).
- **Access control** — file permissions decide who can read, write and execute each file. The OS enforces these.
- **Authentication** — verify that a user is who they claim to be. **Something you know** (password), **something you have** (security token, phone), **something you are** (fingerprint, face scan); strongest when combined (two-factor).
- **Authorisation** — once authenticated, decide what the user is allowed to do. Driven by roles or per-user privileges.
- **Audit logging** — record who did what and when; useful for detecting and investigating breaches.
- **Backups** — keep copies of data so even if the primary copy is lost or corrupted, recovery is possible. Keep some backups **off-site** in case of fire or physical theft.
- **Physical security** — locked server rooms, cable locks on laptops.

## Data integrity

Data has **integrity** when it is **accurate and complete**, with no accidental errors introduced during entry, storage or transfer.

Two complementary techniques: **validation** (catch bad data before it gets stored) and **verification** (confirm that data is correctly entered or transferred).

### Validation — does the data make sense?

Validation checks that data falls within sensible rules. Done by the program automatically. Common validation methods:

- **Range check** — value must be within a specified range. E.g. a month must be 1–12; an exam score must be 0–100.
- **Length check** — string must have the right number of characters. E.g. a UK postcode is 5–7 characters; a credit-card number is 16 digits.
- **Type / character check** — data must be the right type. A telephone-number field accepts only digits (and perhaps `+ -`).
- **Format check** — value must match a pattern. E.g. an email must contain `@`; a date must be `dd/mm/yyyy`.
- **Presence check** — required fields must not be empty.
- **Check digit** — an extra digit appended to a number, computed from the other digits by a formula (e.g. the Luhn algorithm for card numbers, the ISBN-10/13 check). Spots transcription errors. To check, recompute the check digit from the rest and compare with the stored value.
- **Lookup check** — value must be one of a known list (e.g. country code).
- **Consistency check** — value must be consistent with another field (e.g. delivery date $\geq$ order date).

Validation catches **plausibly-formatted but invalid** data — but does **not** catch data that is the right format but factually wrong ("Bob" typed where "Bib" was meant).

### Verification — was the data entered or transferred correctly?

Verification checks that data hasn't been altered (mistyped, corrupted) in moving from one place to another.

#### Verification during data entry

- **Double entry** — the user types the same value twice (e.g. setting a new password). The two values are compared; if they differ, the entry is rejected.
- **Visual verification** — the user reads back what was entered and confirms. The user is the verifier.

#### Verification during data transfer

When data is sent across a network or copied to another medium, transmission errors can flip bits. Use a check sent alongside the data:

- **Parity check** — each byte gets an extra **parity bit**. In **even parity**, the parity bit is set so the total number of 1s in the byte is even; in **odd parity**, odd. The receiver counts the 1s and checks the parity. Detects single-bit errors but not all multi-bit errors.
- **Checksum** — the sender computes a **summary value** (e.g. the arithmetic sum of all bytes, with overflow truncated) of the data and sends it alongside. The receiver recomputes the checksum from the received data and compares it with the value received. If they match, the data is likely intact; if not, an error has occurred (and the data is re-requested).
- **Cyclic Redundancy Check (CRC)** — a more sophisticated checksum based on polynomial division. Detects many more types of errors than a simple sum; widely used in network protocols and storage.

Verification by checksum or CRC does **not** prove the data is correct — only that what arrived matches what was sent. It catches transmission errors but not deliberate tampering (which can compute its own valid checksum) — that requires cryptographic signatures.

### Validation vs verification — keep them straight

- **Validation** = is this data **sensible / allowed**?
- **Verification** = was this data **transcribed / transmitted correctly**?

A value can be valid (in range, right format) but still mistyped — only verification catches that. A value can be perfectly transmitted (verification passes) but contain garbage that should never have been accepted — only validation catches that. Use both.
